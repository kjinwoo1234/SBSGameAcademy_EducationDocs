# docs-review — 학습 따라가기 리뷰

**목표:** 초보가 혼자 읽고 따라가며 학습을 이어갈 수 있는지 비판적으로 판정한다.  
**범위:** 사용자가 이번 호출에서 지정한 경로·파일만 **채점·수정 대상**. 챕터형이면 같은 과정 목차의 **대상보다 앞인 장 전부**와 **뒤인 장 전부**를 범위 대조용으로 읽는다. **직전·직후만으로 끝내지 않는다.** 범위 없으면 되묻는다.  
**하지 말 것:** 대상 밖 파일 수정, 리뷰 대신 전면 재작성(요청 없으면 보고만), 다른 과정으로 “맞춰” 누락 판정.

## STEP 0 — 범위 확인

사용자가 준 파일/폴더/글롭만 **채점 대상**으로 연다. 여러 파일이면 **파일마다** 채점표를 따로 쓴다.

## STEP 1 — 문서 읽기

대상 전체를 읽는다. 목차형(과정 README·루트 README)이면 아래 **챕터형 채점** 대신 STEP 4의 목차형 축소표를 쓴다. (목차형은 장 범위 커버 검사 해당 없음.)

챕터형이면 같은 과정 `README.md` **학습 순서 전체**를 기준으로 다음을 **반드시** 연다.

| 읽을 것 | 깊이 | 목적 |
|---------|------|------|
| **앞 장 전부** (목차상 대상보다 이전인 모든 장) | 각 장: 제목·학습 목표·`###` 소제목. 대상 본문에 나온 문법·API가 “앞에서 배웠다”고 보려면 **해당 앞 장 본문에서 실제 등장**을 확인 | 선행 허용 + 이미 다뤄 여기서 필수는 아닌 것 |
| **대상 장** | 전체 | 채점 본문 |
| **뒤 장 전부** (목차상 대상보다 이후인 모든 장) | 각 장: 제목·학습 목표·`###` 소제목 (본문 전체는 필수 아님) | 다른 장이 가져갈 주제 → 이 장에 없어도 됨 / 이 장에 넣으면 중복·선행 위반 |

**금지:** 직전 1장·직후 1장만 보고 범위 커버·선행을 예로 판정하지 않는다. 중간·멀리 있는 장에 이미 있거나 나중에 다룰 개념을 놓치면 Q2/F5 오판이 난다.

첫 장이면 앞 장 집합 없음. 마지막 장이면 뒤 장 집합 없음 → 이 장에서 과정상 남은 자리인지 본다.

**금지:** 채점·개선 제안 때문에 **다른 과정** 챕터를 열어 “그쪽 순서·목차에 맞추라”고 하지 않는다. 외부 커리큘럼·권위 문서로 “보통 넣는 항목”을 **필수 누락**으로 올리지 않는다. 사용자가 다른 과정·외부 대조를 요청한 경우만 예외.

### STEP 1b — 장 범위 카드 (챕터형 필수)

채점 전 요약(채점표 Q2 근거·`## 범위 대조`에 넣음).

1. **이 장 자리:** 파일명·챕터 제목 + 학습 목표가 가리키는 핵심.
2. **이미 앞 장(전체):** 목차상 **이전 장 어디든** 이미 다뤄, 여기서 재정의·재실습이 **필수는 아닌** 것.
3. **뒤 장으로 미룸(전체):** 목차상 **이후 장 어디든** 제목·목표·소제목이 가져갈 것이라 이 장 본문에 **없어도 되는** 것.
4. **이 장에 있어야 함:** (1)에서 (2)(3)을 뺀 나머지.  
   - 과정 목차 흐름상 **이 주제가 담당할 다리**인데, 앞 장 전체에도 없고 뒤 장 전체에도 안 넘겨진 개념 → 누락.  
   - 직전·직후만의 “사이”가 아니라, **과정 전체에서 이 장만의 빈자리**를 본다.

애매하면 **아니오(Q2)** + 문제점에 “왜 이 장 자리인지 / 앞 장 집합·뒤 장 집합 어디에 있는지(또는 없는지)”를 적는다. 추측으로 타 과정 내용을 채우지 않는다.

## STEP 2 — 비판 렌즈 (채점 전)

다음을 가정하고 읽는다. 하나라도 막히면 감점·문제로 올린다.

1. 비전공·생초보가 **다른 자료 없이** 이 문서만 본다. (과정 목차 순서대로 앞 장까지는 읽었다고 가정)
2. 코드/명령을 **그대로 타이핑·실행**해 본다.
3. **앞 장까지 나오지 않은 문법·API**가 본문·예시·연습에 쓰이면 막힌다. (예외는 아래 「선행 문법」패턴)
4. 앞 장에서 안 배운 말이 설명 없이 나오면 막힌다.
5. “나중에 배운다”만 있고 **지금 고칠 곳**이 안 보이면 막힌다.
6. 연습문제를 풀 때 본문에 없는 개념이 필요하면 막힌다.
7. **장 범위 빈칸:** STEP 1b의 “이 장에 있어야 함”이 본문·예시·연습에 **없으면** 막힌다. 뒤 장 **전체** 중 어디든 가져갈 것·앞 장 **전체** 중 어디든 이미 충분하면 빈칸이 아님.

막연한 칭찬·총평만 쓰지 않는다. **위치(소제목·예시·문제 번호) + 왜 막히는지 + 어떻게 고칠지**를 적는다.

## STEP 3 — 품질 기준 (이 저장소 챕터 표준에서 추출한 패턴)

기준은 **따라가기 쉬운 입문 챕터가 실제로 하는 일**이다. 특정 파일 경로를 인용하거나 링크로 대지 않는다.

### 구조 패턴

- 제목: `# Chapter NN 제목`
- `## 학습 목표` — 2~4개, **할 수 있다/구분한다**처럼 행동·판정 가능
- `## 본문` + `### NN-X` 소제목이 학습 순서
- 하단 `---` 와 상위 목차로 돌아가는 상대 링크
- **`## 세부 주제`·`## 실습 체크리스트` 두지 않음.** 학습 순서는 본문 소제목(`### NN-X`)만 담당한다.

### 따라가기 패턴

- 핵심 용어는 **굵게** 잡은 뒤 **같은 단락/소절에서 한 줄 정의**
- 코드·명령 예제 **직전**에, 학습자가 이어서 **읽고·타이핑·실행·확인**하면 된다는 뜻이 전달되는 안내가 있다.  
  고정 문구(“지금부터 … 해 보겠습니다”)는 **필수가 아니다**.  
  자연스러운 유도면 충분하다. 예: “아래 코드를 `Main` 안에 넣고 실행해 보세요”, “이어서 줄바꿈 차이를 확인해 봅시다”, “같은 뼈대에서 출력만 바꿔 봅니다”.  
  **실패:** 설명·표 다음에 예제가 **갑자기** 나타나고, 지금 따라 할 단계인지 읽을 단계인지 분간이 안 됨
- API는 이름만이 아니라 **`이름("인자")` 사용형** + 괄호·따옴표·`;` 역할 한 줄
- 지금 안 배우는 뼈대 키워드(`using`, `class`, `static`, `void` 등)는  
  **과정 안에서 뼈대를 처음 도입하는 장**에만 **「지금은 뜻을 외울 필요 없음 / 기본 형태만」** + 고칠 위치(`Main`의 `{ }` 안)를 둔다.  
  **이미 앞 장에서 말한 뒤**에는 같은 안내를 **반복하지 않는다.** (예: C# `01-output` 이후 장)
- **이 장 범위**를 문장으로 닫고, 이어질 주제는 “다음 장”으로 미룸 (범위 밖 심화 금지)
- **선행 문법만 사용 (필수):** 본문·예시·연습·정답은 과정 목차 기준 **이전 챕터까지 + 현재 챕터에서 가르치는 문법**만 쓴다.  
  **금지:** 아직 안 나온 `if`/`class`/`async` 등을 “나중에 배움”만 붙이고 본문에 본격 사용.  
  **예외(어쩔 수 없는 경우만):** 콘솔 프로그램 **기본 뼈대**처럼 실행에 꼭 필요한 템플릿. **첫 도입 장**에서만 뜻을 가르치지 말고 **「지금은 형태만」+ 고칠 위치**를 명시. 이후 장은 예제에 뼈대가 나와도 **같은 안내를 되풀이하지 않음**.  
  예외로도 부족하면 예제를 **이미 배운 문법만으로 다시 쓰거나**, 해당 문법을 **현재 장에서 먼저 정의**한 뒤 쓴다.
- **장 범위 커버 (필수):** 같은 과정 목차의 **앞 장 전부·뒤 장 전부**를 대조해, **이 장 자리의 핵심**이 본문·예시·연습에 빠지지 않았는지 본다. **직전·직후만 보면 실패(공정 위반).**  
  **실패:** 제목·학습 목표·과정 전체에서 이 장만의 빈자리가 가리키는 개념이 본문에 없거나, “다음 장”으로만 미루고 지금 장이 비어 있음.  
  **실패 아님:** 이전 장 **어디든** 이미 충분히 다룬 것의 미반복, 이후 장 **어디든** 가져갈 심화의 미포함, 사용자가 이 장에서 빼라고 **명시한** 항목.  
  **금지:** 다른 과정·외부 실라버스로 “빠졌다”고 단정.
- **조건문 입문 챕터:** 예약어 조합(`if`만 / `if`+`else` / `if`+`else if` / `else if` 복수 / 전체+`else` / `switch`+`case`)을 **소절·예제로 단계 분리**했는지 본다. 한 소절·비교 표만으로 `if/else if/else`와 `switch`를 뭉뚱그리면 따라가기 감점·문제로 올린다. (`project-domain` 조건문 챕터 절)

### 예시 패턴

- 실행 가능한 **완결 예제** + 언어 태그(`csharp`, `c`, `cpp` 등 실제 언어)
- 화면·출력이 핵심이면 **`예상 출력`** (`text` 블록)
- 예제 아래 **`코드 해석`**(또는 명령/절차 해석)으로 줄 단위 연결
- 대안 비교(`Write` vs `WriteLine` 등)는 **표** (형태 / 차이 / 이럴 때)
- 한 예제당 핵심 1~2개. 같은 뼈대를 반복해 학습자가 “어디만 바꾸면 되는지” 보이게
- **개념 분리 = 예제 분리 + 설명 분리.** 예제 블록을 나눴으면 안내 문단·코드 해석도 그 블록에만 맞게 둔다. 공통 설명에 다음 예제 개념을 미리 넣거나, 선언 예제에 길이(`.Length`/`sizeof`)·예외·다른 문법을 끼워 넣으면 감점·문제로 올린다. (`project-domain` 예제·설명 분리 절)

### 연습 패턴

- `### 연습문제` — 가능하면 3단계(기초 → 응용 → 확인)
- 문제마다: **문제 / 입력 / 출력 / 조건**
- `### 정답 포인트` — 핵심 한 줄·순서. 본문·연습과 모순 없음

### 문장·밀도 패턴

- 소제목 본문은 **문단 중심**. 불릿만 길게 이어지지 않음 (연습·절차 나열 블록은 예외)
- **문장 중간 괄호 부연 지양:** `차이를(A vs B)`, `2(종료)`, `n--(지금 값…)`처럼 흐름을 끊는 끼워넣기 금지. 풀어 쓴 문장으로. 코드·표·필수 문법 표기 괄호는 OK
- 학습 목표에 나온 용어·기능이 본문에 **다시 등장·설명**
- placeholder(`TODO`, `설명 추가`, `코드 추가`) 없음
- 학습 문서에 **외부 URL/하이퍼링크**로 권위 자료를 심지 않음 (상대 경로 상위 링크는 OK)

## STEP 4 — 채점 (예/아니오 → 100점)

각 항목은 **예 = 만점, 아니오 = 0**. 부분 점수 없음. 애매하면 **아니오** + 문제에 근거 적기.

### 챕터형 (항목당 5점, 20항목 = 100)

| ID | 기준 (예여야 함) | 점 |
|----|------------------|----|
| S1 | `# Chapter NN …` 형식 제목 | 5 |
| S2 | `## 학습 목표` 2~4개, 행동/판정 가능한 문장 | 5 |
| S3 | `## 본문` + `### NN-X` 소제목으로 순서 제시. **`## 세부 주제`·`## 실습 체크리스트` 없음** | 5 |
| S4 | 하단 `---` + 상위 README(또는 동등 목차) 상대 링크 | 5 |
| F1 | 새 핵심 용어가 같은 소절에서 정의됨 | 5 |
| F2 | (코드/명령 예제가 있으면) 블록 직전에 **따라 하기 유도**가 있음. 고정 문구 불필요. 자연스러운 유도면 예. 예제만 덩그러니면 아니오 | 5 |
| F3 | 핵심 API·명령이 사용형으로 나오고 기호(따옴표·괄호·`;` 등) 역할 설명 | 5 |
| F4 | 뼈대 키워드: **첫 도입 장**이면 “지금은 형태만”류 안내 있음. **앞 장에서 이미 안내했으면** 반복 없음 = 예(해당 없음). 뒤 장에 또 넣으면 아니오(군더더기) | 5 |
| F5 | **선행 문법 준수:** 본문·예시·연습에 뒤 장 전용 문법이 없음(불가피 뼈대는 「지금은 형태만」). 장 범위 밖 심화가 실습을 막지 않음 | 5 |
| E1 | 검증 가능한 완결 예시 ≥1 + 올바른 언어/형식 태그 | 5 |
| E2 | 출력이 핵심인 개념에 `예상 출력`(또는 동등한 기대 결과) 있음 | 5 |
| E3 | 예시 뒤 코드/절차 해석 블록 있음 | 5 |
| E4 | 비교할 대안이 있으면 표로 용도·차이 정리 (대안 없으면 **예**) | 5 |
| E5 | 예시가 한 번에 핵심 1~2개에 집중 | 5 |
| P1 | 연습문제 존재 (권장 3문제, 최소 1) | 5 |
| P2 | 각 문제에 문제·입력·출력·조건(해당 없으면 명시) | 5 |
| P3 | 정답 또는 정답 포인트가 문제와 대응 | 5 |
| Q1 | 본문 소제목이 문단 중심 (불릿벽 아님; 연습/절차 제외) | 5 |
| Q2 | **학습 목표 + 장 범위 커버:** (a) 학습 목표 용어·기능이 본문에 등장·설명됨. (b) STEP 1b 기준 **이 장에 있어야 할 핵심**이 본문·예시·연습에 빠지지 않음(**앞 장 전부·뒤 장 전부** 대조, 직전·직후만이 아님). 둘 중 하나라도 실패면 아니오 | 5 |
| Q3 | placeholder/TODO 없음 + 학습 문서에 외부 권위 URL 없음 | 5 |

**예제/코드가 본질이 아닌 순개념 챕터:** F2·E1~E3은 “절차·표·의사코드 등 동등 검증물”이 있으면 예.  
**뼈대 키워드가 아예 안 나오거나, 앞 장에서 이미 “형태만/Main 안만”을 다뤘으면** F4는 예(해당 없음 = 통과). **첫 도입이 아닌데 또 “지금은 형태만”을 넣으면** F4 아니오.  
**선행 문법:** 과정 README 순서로 **이전 장 전부**를 대조한다. 애매하면 아니오(F5) + 문제에 미학습 문법명을 적는다.  
**장 범위 커버:** **앞 장 전부·뒤 장 전부**를 안 읽고(또는 직전·직후만 보고) Q2를 예로 주지 않는다. 누락이 있으면 문제점에 **빠진 핵심 / 왜 이 장 자리인지 / 앞·뒤 집합 어디에 없는지**를 적고, 개선안에 **넣을 소절·예제 요지**를 적는다. 심화 욕심으로 뒤 장 범위를 끌어오면 F5·범위 밖 심화로 따로 지적.

## STEP 5 — 보고 형식 (필수)

한국어. 사용자 대면 설명은 저장소 caveman 문체를 따르되, **채점표·문제·개선안 본문은 오해 없이 완전한 문장**으로 쓴다 (점수·경로·기준 ID는 정확).

### 산출물 위치 (필수)

리뷰 **전체 본문**은 채팅에만 두지 말고, 저장소 루트 **`.result/`** 아래 마크다운 파일로 쓴다.

| 항목 | 규칙 |
|------|------|
| 디렉터리 | 저장소 루트 `.result/` (없으면 생성). `docs/`·`.cursor/`에 넣지 않음. cwd와 무관하게 **repo root** 기준 |
| 파일명 (단일) | `review-<상대경로-슬래시대신하이픈>.md` — 예: `docs/csharp/01-output.md` → `.result/review-docs-csharp-01-output.md` |
| 파일명 (여러 파일) | 파일마다 위 규칙으로 각각 저장. 추가로 `.result/review-summary-<YYYYMMDD-HHmm>.md`에 총점·최저점만 요약 |
| 덮어쓰기 | 같은 대상 재리뷰면 **같은 파일명에 덮어쓴다** (최신만 유지) |

채팅에는 **총점·한줄 판정·저장 경로**만 짧게. 상세 채점표·문제·개선안은 `.result/*.md`가 원본.

```markdown
# 교육 자료 리뷰: <파일명 또는 범위>

## 총점: <N>/100

## 한줄 판정
<초보 혼자 따라가기: 가능 / 조건부 / 어려움 — 이유 1문장>

## 범위 대조 (챕터형)
- 앞 장 전부: <목차상 이전 장 목록 또는 `01~NN` 요약; 없음>
- 뒤 장 전부: <목차상 이후 장 목록 또는 `NN~마지막` 요약; 없음>
- 이 장에 있어야 함(요약): <핵심 몇 개>
- 빠진 핵심: <없음 | 목록>
- (참고) 직전·직후만 본 경우 → Q2 예 불가(공정 위반)

## 채점표
| ID | 결과 | 점수 | 근거 (한 줄, 위치 포함) |
|----|------|------|-------------------------|
| S1 | 예/아니오 | 5/0 | … |
| … | … | … | … |

## 문제점
1. **[위치]** <구체적 문제>. <왜 학습을 막는가>.
2. …

## 개선안
1. **[위치]** <바로 적용 가능한 수정>. <기대 효과 한 줄>.
2. …

## 잘된 점 (선택, 최대 3)
- …
```

- 문제점·개선안은 **우선순위** 순 (따라가기 차단·**장 범위 누락** > 혼란 > 문체).
- 총점 90+여도 비판적으로 **남는 리스크**가 있으면 문제점에 적는다. 없으면 “중대 문제 없음” 한 줄.
- 여러 파일이면 파일별 섹션 후, 맨 아래 **범위 평균·최저점 파일**만 요약.
- 목차형 리뷰는 `## 범위 대조` 생략.
