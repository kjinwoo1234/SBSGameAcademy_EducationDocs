# review-improve — 리뷰 개선안 반영 + 점수 루프

**목표:** `.result/` 리뷰의 **개선안**을 대상 `docs/`에 외과 적용한 뒤, **총점 > 95**가 될 때까지 **개선 → `/review`(docs-review) → 개선 → …** 를 같은 호출 안에서 반복한다.  
**전제:** 품질 패턴·채점 기준은 `docs-review`와 같다. 난이도·톤을 임의로 낮추지 않는다.  
**하지 말 것:** 리뷰에 없는 “顺便 리팩터”, 범위 밖 파일 수정, 학습 문서에 외부 URL 삽입.

**점수 목표:** 총점 **> 95**. 챕터형 채점은 항목당 5점이라 가능한 점수는 5 단위 → 실질 통과는 **100**.

## STEP 0 — 입력 확인

다음 중 하나로 대상을 정한다. Ambiguous면 되묻는다.

| 사용자가 준 것 | 할 일 |
|----------------|--------|
| 문서 경로만 (`docs/...`) | `.result/review-<상대경로-하이픈>.md` 찾기. 없으면 **같은 호출에서** `docs-review`로 1회 채점·저장 후 루프 진입 |
| 리뷰 경로만 (`.result/review-....md`) | 리뷰 제목·본문에서 문서 경로 추출 |
| 둘 다 | 그대로 사용. 불일치하면 확인 |
| 여러 파일 | **파일마다** 아래 루프를 **각각** 완주(파일 A가 >95 된 뒤 파일 B) |

저장소 루트 기준. `.result/` 없으면 생성해도 됨(리뷰·improve 기록용).

**범위 제한 (사용자 명시 시):** “N번만”, “F2만”, “표만”, “루프 없이 1회만” → 그 범위만. 루프 없이면 STEP 7 생략·목표 점수 미적용.

## STEP 1 — 리뷰 파싱

리뷰 마크다운에서 필수 추출:

1. **대상 문서** (`# 교육 자료 리뷰: …`)
2. **총점·한줄 판정** (보고·루프 판정용)
3. **`## 범위 대조`** (있으면) — 빠진 핵심·이 장에 있어야 함. Q2 개선 시 경계로 사용
4. **`## 개선안`** 목록 — **적용 큐**. 번호·`[위치]`·수정 내용
5. **`## 문제점`** — 개선안이 모호할 때 의도 파악용
6. **채점표에서 아니오인 ID** — 개선 후 그 항목이 해소되는지 점검용

`## 잘된 점`은 **읽기만**. 건드리지 않음.

**이미 총점 > 95**이면 문서 수정 없이 “목표 달성(수정 없음)”만 보고하고 종료.

## STEP 2 — 적용 우선순위

개선안 번호 순을 기본으로 하되, 충돌 시:

1. 따라가기 차단 (유도 문장, 깨진 예시, 모호한 연습 출력)
2. **장 범위 누락** (같은 과정 **앞·뒤 장 전부** 대비 이 장 핵심) · 학습 목표 ↔ 본문·연습 정렬
3. 비교 표·코드 해석 보강
4. 문체·미세 표기

사용자가 “N번만”, “F2만”, “표만”이면 그 범위만.

## STEP 3 — 분기 개선안 (A 또는 B)

개선안에 **둘 중 하나**가 있으면:

| 기본 선택 | 이유 |
|-----------|------|
| 내용을 **유지·보강**하는 쪽 (예: 학습 목표에 항목 추가) | 이미 본문·연습에 있는 개념을 지우면 학습 공백 |
| 사용자가 명시하면 그 선택 | “다음 장으로 미뤄”, “삭제” 등 |

선택한 분기와 **버린 분기**를 STEP 5 보고에 한 줄 적는다.

## STEP 4 — 문서 수정 (외과)

1. 대상 `docs/` 파일 **전체**를 읽는다.
2. 개선안마다 **위치**에만 최소 삽입·수정.
3. `docs-review` 패턴 유지:
   - 따라 하기 유도는 자연스러운 한 줄이면 됨 (고정 문구 필수 아님)
   - 새 용어는 같은 소절에서 정의
   - 비교는 표 우선
   - 연습 출력은 자가채점 가능하게 구체화
   - placeholder·외부 권위 URL 넣지 않음
   - **문장 중간 괄호 부연**으로 설명을 끼우지 않음 (`project-domain` 본문 문장·`docs-review` 문장 패턴)
   - **예제·설명 분리:** 개선 시 개념을 한 예제에 다시 뭉치지 않음. Length/`sizeof`·예외는 각자 블록 (`project-domain`·`docs-review` 예시 패턴)
   - **선행 문법만:** 수정 후에도 뒤 장 문법을 새로 끌어오지 않음. 미학습 문법이 있으면 이미 배운 문법으로 바꾸거나, 현재 장에서 짧게 정의한 뒤 씀. 뼈대 예외의 「지금은 형태만」은 **첫 도입 장만** — 앞 장에 있으면 뒤 장에 **추가하지 않음** (`project-domain`·`docs-review` F4)
   - **장 범위 커버:** 리뷰 `## 범위 대조`·Q2 개선안이 가리키는 **이 장 핵심**만 보강. 이후 장 **어디든** 가져갈 범위를 끌어와 채우지 않음. 이전 장 **어디든** 이미 있는 내용의 중복 추가 금지 (`docs-review` STEP 1b·Q2). 직전·직후만 보고 판단한 리뷰면 재리뷰 시 **전부** 다시 대조.
   - **`## 세부 주제`·`## 실습 체크리스트`가 있으면 삭제** (본문 `### NN-X`만 남김). 한 파일에서 발견하면 같은 과정 `docs/<과정>/`에 잔여 없는지 확인; 사용자 보정·재지적이면 `instruction-ecosystem` 보정 루프(전수+지침).
4. 리뷰에 없는 인접 “개선”은 하지 않음. 단, 개선안 적용에 **필수인 한 줄**(예: 표 헤더)은 허용.
5. 여러 개선안이 같은 위치를 건드리면 **한 번의 편집**으로 합친다.
6. **다른 과정 문서로 맞춰 고치지 않음.** 대상·같은 과정 선행 장·해당 리뷰만. 사용자가 다른 과정 참고를 명시할 때만 연다. (`project-domain` 작업 규칙 8)

## STEP 5 — 라운드 보고 (짧게)

### 채팅 (라운드마다 짧게, 최종은 STEP 7)

- 라운드 번호·적용한 개선안 번호·한 줄씩
- 건너뛴 것(없으면 “없음”) + 이유
- 수정 파일 경로
- 재리뷰 총점(STEP 7 후)

### 기록 파일 (필수, 루프 종료 시 최종본 덮어쓰기)

`.result/improve-<상대경로-하이픈>.md`에 남긴다.

```markdown
# 리뷰 반영: <문서 경로>

- 근거 리뷰(최종): `.result/review-….md`
- 반영 시각: <ISO 또는 로컬 시각>
- 최종 총점: <N>/100
- 라운드 수: <k>
- 종료 사유: 목표 달성(>95) | 상한 | 정체 | 사용자 중단 | 루프 없음

## 라운드 요약
| 라운드 | 적용 전 총점 | 적용 후 재리뷰 총점 | 적용한 개선안 |
|--------|--------------|---------------------|---------------|
| 1 | … | … | … |

## 적용 (최종 라운드 기준 또는 누적)
1. [개선안 N] <한 줄>
2. …

## 분기 선택
- 개선안 N: <선택한 쪽> (버린 쪽: …)

## 건너뜀
- … / 없음

## 수정 파일
- `docs/…`
```

## STEP 6 — 검증 (라운드)

- 수정 후 파일을 다시 읽어, 개선안 위치가 **실제로** 바뀌었는지 확인.
- 채점표에서 아니오였던 ID가 이번 수정으로 풀릴 만하면 라운드 메모에 `해소 예상: F2, E4`처럼 적음.

## STEP 7 — 점수 루프 (기본 ON)

한 라운드 = **STEP 1~6(개선)** → **`docs-review` 전체 공정으로 같은 문서 재채점** → `.result/review-<…>.md` 덮어쓰기 → 총점 판정.

```
루프:
  1. 현재 리뷰 파싱 (STEP 1)
  2. 총점 > 95 → 성공 종료
  3. 개선안 없음 → 정체 종료 (채팅에 잔여 아니오 ID·사유)
  4. STEP 2~6 적용
  5. skills/docs-review/SKILL.md 대로 재리뷰·`.result/review-*.md` 저장
  6. 라운드 수 +1, STEP 5 기록 갱신
  7. 총점 > 95 → 성공 종료
  8. (직전 재리뷰 총점과 동일)이 2회 연속 → 정체 종료
  9. 라운드 수 ≥ 5 → 상한 종료 (현재 총점·남은 개선안 보고)
  10. 아니면 1로
```

| 규칙 | 내용 |
|------|------|
| 목표 | 총점 **> 95** (실질 **100**) |
| 재리뷰 | **반드시** `docs-review` STEP을 따른다. 채점표·문제점·개선안 형식 유지. 채팅은 총점·한줄·경로만, 상세는 `.result/` |
| 상한 | 개선→재리뷰 **최대 5 라운드** (무한 루프 금지) |
| 정체 | 개선안 고갈, 또는 재리뷰 총점이 **2회 연속 동일** |
| 중단 | `stop review-improve` / 범위 철회 시 즉시 중단, 그때까지의 improve 기록 남김 |
| 단독 `/review` | 사용자가 `/review`만 호출하면 이 루프를 돌리지 않음. 점수만 |

**금지:** 점수를 올려 보이려고 채점 기준을 느슨하게 적용하거나, 리뷰에 없는 대규모 재작성으로 “통과용” 문서 만들기.

## 경계

- 과정 README 목차·파일 rename이 필요해지면 **보고만** 하고, 사용자 확인 전에 목차를 마음대로 바꾸지 않음.
- `stop review-improve` / 범위 철회 시 추가 수정·재리뷰 중단.
- 루프 중 재리뷰는 `/review-improve` 공정의 **일부**다. 사용자가 `/review`만 요청한 경우에는 여전히 채점·보고만.
