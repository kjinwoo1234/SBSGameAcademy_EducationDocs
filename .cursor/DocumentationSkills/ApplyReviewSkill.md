# ApplyReviewSkill

`AIStudentReviewSkill`, `CurriculumReviewSkill`, `DocumentReviewSkill`의 리뷰 결과를 실제 문서 개선 작업으로 전환할 때 사용하는 통합 실행 가이드입니다.

## 1) 목적
- 리뷰 문서를 "분석 보고서"에서 "실제 문서 수정"으로 연결한다.
- 학생 관점/문서 품질/커리큘럼 구조 관점을 함께 반영해 개선 우선순위를 정한다.
- 변경 후 재검토까지 포함한 개선 루프를 표준화한다.

## 1-1) 적용 범위 선언 (실행 시작 시 필수)
- 실행 시작 시 반드시 아래를 먼저 명시한다.
  - 이번 배치 대상 트랙/문서
  - 제외 범위(이번 배치에서 미반영 문서)
  - 완료 기준(예: 템플릿 3개 문서 적용 완료)
- 범위를 명시하지 않으면 대규모 변경에서 누락/중복이 쉽게 발생한다.

## 2) 입력 문서
- 학생 관점 리뷰: `./.result/ReviewResult-student.md`
- 문서 품질 리뷰: `./.result/ReviewResult-document.md`
- 커리큘럼 리뷰: `./.result/ReviewResult-curriculum.md`
- 통합 리뷰(있는 경우): `./.result/SkillIntegratedReviewResult.md`

리뷰 문서가 일부만 있어도 실행 가능하지만, 가능한 3종 모두를 먼저 반영한다.
- `ReviewResult-curriculum.md`가 없으면, 우선 학생/문서 리뷰 기반으로 1차 개선을 실행하고 이후 커리큘럼 리뷰를 생성해 2차 구조 개선을 수행한다.

## 3) 적용 원칙
1. **타당성 검토 우선**
   - 각 리뷰의 이슈가 실제 문서에서 확인되는지 원문으로 재확인한다.
2. **근거 기반 수정**
   - 모든 수정 항목은 "문제-영향-수정-예상 효과"를 1줄 이상 기록한다.
3. **작은 배치 적용**
   - 대규모 일괄 수정 대신 트랙/섹션 단위로 나눠 반영한다.
4. **학생 경험 우선**
   - 점수 개선보다 "학습자가 실제로 덜 막히는가"를 우선 판단 기준으로 둔다.

## 4) 리뷰 통합 방식
중복/충돌 이슈는 아래 규칙으로 병합한다.

- 공통으로 반복된 이슈: **즉시 반영 1순위**
- 학생 관점에서 심각(이탈 위험) + 문서 품질도 낮음: **최우선**
- 커리큘럼 구조 이슈(문서 신설/순서 개편): **2순위**
- 표현 다듬기/문체 개선: **후순위**

충돌 시 우선순위:
1) 학습 실패를 줄이는 수정  
2) 검증 요소(체크리스트/퀴즈) 보강  
3) 구조/표현 정리

## 5) 실행 절차 (필수)
### Step 1. 이슈 수집
- 각 리뷰의 `주요 문제점`, `상세 개선안`, `다음 액션`을 모아 통합 리스트를 만든다.
- 동일 이슈를 묶어 중복 항목을 제거한다.

### Step 2. 작업 분류
- 분류 A: 즉시 수정 가능한 문장/섹션 추가
- 분류 B: 다수 문서 템플릿 표준화
- 분류 C: 신규 문서 신설(브릿지/미니프로젝트)

### Step 3. 우선순위 부여
각 작업에 아래 태그를 붙인다.
- 영향도: `상/중/하`
- 작업비용: `상/중/하`
- 범위: `단일 문서/트랙/전체`

권장 실행 순서:
1. 영향도 상 + 작업비용 하  
2. 영향도 상 + 작업비용 중  
3. 나머지

### Step 3-1. 배치 단위 결정
- 한 번에 과도한 파일을 수정하지 않는다.
- 권장 배치 크기:
  - 소규모: 1~5 파일
  - 중규모: 6~15 파일
  - 대규모: 16 파일 이상(권장하지 않음, 분할 필요)
- 대규모 작업은 "대표 문서 -> 트랙 확장 -> 전체 확장" 3단계로 분리한다.

### Step 4. 문서 수정
- 실제 파일을 수정하고, 링크/섹션/용어 일관성을 함께 점검한다.
- 템플릿 확장은 대표 문서(입문/중간/마무리)부터 적용 후 전체 확장한다.

### Step 5. 검증
- 수정 파일 기준으로 오탈자/링크/섹션 누락을 확인한다.
- 기존 리뷰 이슈가 실제로 해소되었는지 재확인한다.
- 사용자 규칙(체크박스 금지, 개념/용어/예제 포함)을 위반한 신규 문장이 없는지 확인한다.
- `- [ ]`, `[ ]` 패턴이 재유입되지 않았는지 정규식으로 확인한다.

### Step 6. 재평가
- 동일 스킬로 재리뷰해 점수 변화를 기록한다.
- 필요하면 `ReviewResult-*` 파일을 최신화한다.

## 6) 문서 개선 확인 항목
1. README에 시작 가이드/진행 구조가 포함되어 있는가?
2. 각 트랙 README에 권장 대상/선수지식/완주 후 역량이 포함되어 있는가?
3. 본문 문서에 학습 목표/실습/검증 요소가 포함되어 있는가?
4. 문서 간 이전/다음/상위 링크가 자연스러운가?
5. 공통 브릿지(디버깅/협업)가 트랙 문서와 연결되어 있는가?
6. 학생 관점 이상점(최소 3개)이 실제로 해소되었는가?

## 7) 결과물 작성 규칙
- 개선 실행 로그는 `./.result/SkillIntegratedReviewResult.md`에 누적 정리한다.
- 스킬별 재리뷰 결과는 기존 파일명 규칙을 유지한다.
  - `ReviewResult-student.md`
  - `ReviewResult-document.md`
  - `ReviewResult-curriculum.md`
- 실행 로그에는 "이번 배치에서 실제 수정한 파일 목록"과 "미반영 사유"를 반드시 함께 기록한다.

## 8) 출력 템플릿 (실행 결과 보고)
아래 구조를 따른다.

1. 적용한 리뷰 문서 목록
2. 통합 이슈 목록(중복 제거)
3. 우선순위 매트릭스(영향도/작업비용)
4. 실제 반영한 파일 목록
5. 반영 내용 요약(문제-수정-효과)
6. 미반영 항목 및 사유
7. 재평가 점수 변화
8. 다음 액션(1~3)

## 9) 사용자 규칙
학습자료 문서 작성/개선 시 반드시 적용할 사용자 규칙이다.

1. 아래 규칙을 따르지 않는 문서는 우선 수정 대상으로 분류한다.
2. `- [ ]` 형태의 체크박스 항목은 사용하지 않는다.
3. 학습 자료는 문서 자체만으로 학습이 가능해야 한다. 개념 설명, 용어 정의, 구체적 활용 방법, 예제를 포함한다.
4. 문법/명령어/구체적 사용법은 반드시 문서 본문에 포함한다.
5. 점검 질문(이해 점검/자기 점검)에 대한 답을 학습자료 본문에서 찾을 수 있어야 한다.
6. 문서는 "요약"에 그치지 않고, 실습 가능한 수준의 상세도를 유지한다.
7. 불필요하거나 중복된 내용 또는 파일은 깔끔하게 정리한다.

## 10) 금지/권장 표현 규칙
- 금지: `- [ ]` 또는 `[ ]` 형태 체크박스
- 권장: 질문형 불릿(`- ...할 수 있는가?`) 또는 번호형 점검 항목(`1. ...`)
- 권장: "문제 -> 왜 문제인지 -> 어떻게 고칠지 -> 기대 효과" 4단 구조

## 11) 빠른 실행 예시
1. 리뷰 문서 2~3개를 읽고 공통 이슈 3개를 추린다.
2. 영향도/작업비용 태그를 붙여 1순위 3개를 선택한다.
3. 문서 5~10개 단위로 수정한다.
4. 규칙 위반 패턴(`- [ ]`, `[ ]`)을 검색해 0건 확인한다.
5. `SkillIntegratedReviewResult.md`에 실행 로그를 누적한다.
