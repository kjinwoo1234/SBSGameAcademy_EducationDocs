# C 문서 품질 가이드라인

이 문서는 `docs/c` 하위 문서를 작성/수정할 때 문서 품질을 일정하게 유지하기 위한 공통 기준입니다.

## 1) 문서 기본 구조(필수)
- 모든 챕터 문서는 아래 순서를 유지한다.
  1. `# Chapter XX 제목`
  2. `## 학습 목표` (2~4개)
  3. `## 세부 주제` (README 목차와 번호 일치)
  4. `## 실습 체크리스트` (1~3개)
  5. `## 본문`
  6. 문서 하단 `---` + `[상위 문서로 돌아가기](./README.md)`
- `## 본문`에는 세부 주제와 동일한 소제목(`XX-X`)을 같은 순서로 배치한다.

## 2) 본문 작성 기준(필수)
- 각 세부 주제(`XX-X`)마다 최소 2~4개 설명 bullet 작성
- 설명은 비전공자/생초보자 기준으로 쉬운 표현 사용
- 용어를 처음 쓰면 한 줄로 뜻을 풀어 설명
- "왜 필요한지"와 "어디서 쓰는지"를 함께 적기
- 설명 중간에 placeholder(`설명 추가`, `TODO`, `코드 추가`)를 남기지 않는다.
- 항목이 4개 이상이거나 비교/매핑 정보가 많으면 `-` 나열 대신 표를 우선 사용한다.
- 문서 안에 새로 등장한 개념/단어(예: 레이블, 바이어스, 포인터 역참조)는 같은 섹션에서 바로 정의한다.
- 정의 없이 등장한 개념이 있으면 예시 코드 아래에 "용어 설명" 블록을 추가한다.
- 외래어/영문 용어(예: callback, fall-through, interface, token)는 한국어 뜻을 반드시 함께 적는다.
- 약자/두문자어(예: RAII, OOP, CLI, API, STL)는 첫 등장 위치에서 반드시 풀네임(원문)과 한국어 의미를 함께 적는다.
- 약자 설명은 "약자 -> 풀네임 -> 한국어 의미 -> 왜 필요한지(1줄)" 순서로 작성한다.
- 영어 약자만 단독으로 쓰지 말고, 초보자 기준으로 단어별 핵심 뜻(Resource, Acquisition 등)을 필요 시 함께 풀이한다.
- 새 용어를 추가하기 전에 "이전 챕터에서 이미 설명했는지"를 먼저 확인한다.
- 이전 챕터에서 아직 배우지 않은 용어라면 현재 챕터에 1~2줄 정의를 즉시 추가한다.
- 용어 설명만으로 이해가 어려울 수 있으면, 짧은 예시 코드를 함께 제공한다.
- 특히 배열/포인터/입출력처럼 동작을 눈으로 확인해야 하는 용어는 예시 코드를 우선 추가한다.
- `## 학습 목표`와 `## 세부 주제`에 쓴 핵심 용어는 본문에서 반드시 다시 등장해야 하며, 정의/설명이 있어야 한다.
- 학습 목표/세부 주제에서 먼저 제시한 용어가 본문에서 누락되면, 해당 섹션(`XX-X`)에 "용어 설명" 또는 1~2줄 정의를 추가한다.
- 학습 목표/세부 주제의 용어는 본문 소제목과 1:1 또는 N:1로 연결되도록 작성한다(독자가 "어디에서 배우는지" 바로 찾을 수 있어야 함).
- "우리 과정에서는 ~로 정했다" 같은 운영/정책 문장보다, 학습자 관점의 이유 설명("왜 이것을 먼저 배우는지")을 우선한다.
- 도구/문법 선택을 안내할 때는 결정 자체보다 선택 근거(초보자 관점의 이해 난이도, 실수 가능성, 실무 사용성)를 먼저 제시한다.

## 3) 예시 코드 기준(필수)
- 각 문서에 최소 1개 이상의 컴파일 가능한 C 코드 예시 포함
- 코드 블록 언어 표시는 반드시 `c`
- 초급 문서에서는 한 예제당 핵심 포인트 1~2개만 다룬다
- 필요하면 `예상 출력` 블록(`text`)을 함께 제공한다.
- 예제는 문서 난이도에 맞는 최소 의존 헤더만 사용한다.
  - 단순 출력: `stdio.h`
  - 문자열: `string.h`
  - 동적 메모리: `stdlib.h`
- 예제 코드에 본문에서 아직 설명하지 않은 문법/표현이 나오면, 반드시 바로 아래에 설명을 추가한다.
  - 예: `%%`, `%.2f`, `scanf` 반환값 체크, `&` 사용 이유, 형식 지정자-타입 매핑

## 4) 연습문제/정답 기준(필수)
- 각 문서에 `연습문제`와 `정답` 또는 `정답 포인트`를 포함한다.
- 연습문제는 가능하면 단계형으로 구성한다.
  - 1번: 기초
  - 2번: 응용
  - 3번: 확인/심화
- 정답은 전체 코드 또는 핵심 정답 포인트로 제공한다.
- 연습문제는 문제별 상세 포맷을 기본으로 사용한다.
  - 문제
  - 입력
  - 출력
  - 조건(또는 힌트)

## 5) 난이도/설명 깊이 기준
- 현재 챕터에서 필요하지 않은 심화 개념은 짧게만 언급하고, 학습할 다음 챕터를 연결한다.
- 과도한 이론 나열보다 "실행 가능한 예제 + 오류 방지 팁"을 우선한다.
- 필요성 판단 기준:
  - 지금 이해가 없으면 실습이 막히는 개념: 간단히라도 즉시 설명
  - 지금 없어도 실습 가능한 심화 개념: 다음 챕터 링크만 남기고 최소 설명

## 6) 기술 정확성 규칙(필수)
- 단정 문장을 쓸 때는 예외 가능성을 확인한다.
  - 예: "항상" 대신 "대부분", "보통" 사용
- 수식/비트 설명은 "기호 의미 -> 계산 과정 -> 결과" 순서로 쓴다.
- 오해하기 쉬운 개념(`scanf`의 `&`, 부동소수점 바이어스 등)은 최소 1개 구체 예시를 포함한다.
- "오류 방지 문법/패턴"을 설명할 때는 반드시 3가지를 함께 적는다:
  1) 오류가 실제로 언제 발생하는지(발생 조건),
  2) 왜 문제가 되는지(증상/영향),
  3) 해당 문법이 어떻게 막는지(해결 원리).
- 비교 가능한 대안(`A vs B`)이 있으면 "사용 용도/장점/단점"을 표로 정리하고, 입문자가 먼저 잡을 기준을 한 줄로 명시한다.

## 7) 일관성 규칙
- 챕터 번호/세부 번호 표기 통일 (`Chapter 03`, `03-1` 형식)
- 네이밍, 용어, 말투를 문서 간 일관되게 유지
- 문장 끝 표현은 가능하면 같은 톤으로 정리
- 목차(`docs/c/README.md`)와 파일명/링크가 항상 일치하도록 유지
- 학습 순서가 바뀌면 관련 챕터 둘 다 함께 조정한다.
  - 예: 형식 지정자를 `02`에서 제거했다면 `03`에 반드시 보강

## 8) 초보자 친화 규칙
- 핵심 개념은 "짧은 직관 설명 + 실제 동작 설명" 순서로 작성한다.
- 자주 하는 실수(예: `scanf`의 `&` 누락, 배열 인덱스 범위 초과)를 명시한다.
- 에러 상황에서 무엇을 확인할지 짧은 체크포인트를 제공한다.
- 한 번에 너무 많은 개념을 섞지 않고, 챕터 핵심 주제에 집중한다.
- "어디까지가 현재 챕터 범위인지"를 분명히 적는다.
- 내부 운영 관점 표현(예: 과정 운영 기준, 팀 내부 합의 문장)은 본문에서 최소화하고, 학습자 질문에 직접 답하는 문장으로 바꾼다.
- "무엇을 쓴다"로 끝내지 말고 "언제 오류가 나고, 왜 이 방식이 필요한지"를 먼저 이해시키는 설명 흐름을 유지한다.

## 9) 개발환경/실행 안내 규칙
- 환경별 차이가 큰 명령/함수는 사용 조건을 함께 적는다.
- 필요 시 CLI 실행법과 IDE 실행법을 함께 제공한다.
- 개발환경 설명은 초급 문서에서 STEP BY STEP 형식으로 작성한다.

## 10) 금지/주의 사항
- 실행 불가능한 가짜 코드, placeholder 문구, 잘못된 가짜 정보를 남기지 않는다.
- 프로젝트 스타일과 충돌하는 표기(번호 체계, 링크 형식)를 사용하지 않는다.
- 이미 합의된 학습 순서를 임의로 되돌리지 않는다.
- 사용자가 요청한 난이도/톤(예: 생초보자 기준)을 임의로 낮추지 않는다.

## 11) 작성 후 점검 체크리스트
- `## 본문`이 존재하는가?
- 세부 주제별 설명이 모두 채워졌는가?
- 예시 코드가 실제 컴파일 가능한가?
- 예시 코드에 새로 등장한 문법/기호를 본문에서 설명했는가?
- 본문에 정의 없이 등장한 개념/단어가 없는가?
- 연습문제와 정답(또는 정답 포인트)이 있는가?
- placeholder/TODO 문구가 남아있지 않은가?
- 상위 문서 링크가 올바른가?
- 환경 의존 요소가 있다면 조건/범위를 명시했는가?
- `학습 목표`에 나온 용어가 본문에서 빠짐없이 설명되는가?
- `세부 주제`의 각 키워드가 본문 해당 소제목에서 정의/예시로 연결되는가?
- 약자/영문 용어의 첫 등장 위치에 풀네임과 한국어 의미가 함께 있는가?
- 약자 설명이 "무엇의 약자인지"를 넘어서 학습 맥락(왜 쓰는지)까지 포함하는가?
- 설명 문장이 내부 정책 안내가 아니라 학습자 관점의 이유 설명으로 작성되었는가?
- 오류 방지 문법/패턴에서 "발생 조건 -> 문제 영향 -> 해결 원리" 3단계가 모두 포함되었는가?
- 대안이 있는 주제에서 "사용 용도/장점/단점" 비교 표가 제공되었는가?
