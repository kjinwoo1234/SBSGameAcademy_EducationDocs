# C 문서 품질 가이드라인

이 문서는 `docs/c` 하위 문서를 작성/수정할 때 문서 퀄리티를 일정하게 유지하기 위한 기준입니다.

## 1) 문서 기본 구조(필수)
- 모든 챕터 문서는 아래 순서를 유지한다.
  1. `# Chapter XX 제목`
  2. `## 학습 목표` (2~4개)
  3. `## 세부 주제` (README 목차와 번호 일치)
  4. `## 실습 체크리스트` (1~3개)
  5. `## 본문`
  6. 문서 하단 `---` + `[상위 문서로 돌아가기](./README.md)`
- `## 본문`에는 세부 주제와 동일한 소제목(`XX-X`)을 같은 순서로 배치한다.

## 2) 본문 작성 기준(필수)
- 각 세부 주제(`XX-X`)마다 최소 2~4개 설명 bullet 작성
- 설명은 비전공자/생초보자 기준으로 쉬운 표현 사용
- 용어를 처음 쓰면 한 줄로 뜻을 풀어 설명
- "왜 필요한지"와 "어디서 쓰는지"를 함께 적기
- 설명 중간에 placeholder(`설명 추가`, `TODO`)를 남기지 않는다.

## 3) 예시 코드 기준(필수)
- 각 문서에 최소 1개 이상의 컴파일 가능한 C 코드 예시 포함
- 가능하면 `#include <stdio.h>` 기준의 최소 실행 예제 제공
- 코드 블록 언어 표시는 반드시 `c`
- Visual Studio(MSVC) 학습 흐름에 맞는 예시 우선
- 초급 문서에서는 한 예제당 핵심 포인트 1~2개만 다룬다
- 필요하면 `예상 출력` 블록(`text`)을 함께 제공한다.

## 4) 연습문제/정답 기준(필수)
- 각 문서에 `연습문제`와 `정답` 또는 `정답 포인트` 포함
- 연습문제는 단계형으로 구성
  - 1번: 기초
  - 2번: 응용
  - 3번: 확인/심화
- 정답은 전체 코드 또는 핵심 정답 포인트 제공

## 5) 난이도/설명 깊이 기준
- 현재 챕터에서 필요하지 않은 심화 개념은 짧게만 언급하고, 학습할 다음 챕터를 연결한다.
- 예: 포인터 내부 원리는 "이 챕터에서는 직관만", 자세한 내용은 포인터 챕터에서 설명.
- 과도한 이론 나열보다 "실행 가능한 예제 + 오류 방지 팁"을 우선한다.
- 필요성 판단 기준:
  - 지금 이해가 없으면 실습이 막히는 개념: 간단히라도 즉시 설명
  - 지금 없어도 실습 가능한 심화 개념: 다음 챕터 링크만 남기고 최소 설명

## 6) 일관성 규칙
- 챕터 번호/세부 번호 표기 통일 (`Chapter 03`, `03-1` 형식)
- 네이밍, 용어, 말투를 문서 간 일관되게 유지
- 문장 끝 표현은 가능하면 같은 톤으로 정리
- 목차(`docs/c/README.md`)와 파일명/링크가 항상 일치하도록 유지
- 학습 순서가 바뀌면 관련 챕터 둘 다 함께 조정한다.
  - 예: 형식 지정자를 `02`에서 제거했다면 `03`에 반드시 보강

## 7) 초보자 친화 규칙
- 자주 하는 실수(예: `scanf`의 `&` 누락, 배열 인덱스 범위 초과) 반드시 명시
- 에러 상황에서 무엇을 확인할지 짧은 체크포인트 제공
- 한 번에 너무 많은 개념을 섞지 않고, 챕터 핵심 주제에 집중
- 비전공자 기준으로 비유 1개까지 허용하되, 비유 뒤에 정확한 기술 표현을 함께 적는다.
- "어디까지가 현재 챕터 범위인지"를 분명히 적는다.

## 8) 개발환경/실행 안내 규칙
- 기본 기준은 `Windows + Visual Studio + MSVC(cl)`로 통일한다.
- CLI 실행법과 IDE 실행법이 모두 필요하면 둘 다 제공한다.
  - CLI 예: `cl main.c` -> `.\main.exe`
  - IDE 예: `Ctrl + Shift + B`, `Ctrl + F5`
- 개발환경 설명은 초급 문서에서 STEP BY STEP 형식으로 작성한다.

## 9) 금지/주의 사항
- 실행 불가능한 가짜 코드, placeholder 문구(`TODO`, `코드 추가`)를 남기지 않는다.
- 챕터 범위를 벗어나는 고급 개념을 본문 중심으로 과도하게 설명하지 않는다.
- 프로젝트 전체 스타일과 충돌하는 표기(번호 체계, 링크 형식)를 사용하지 않는다.
- 이미 합의된 학습 순서를 임의로 되돌리지 않는다.
- 사용자가 요청한 난이도/톤(예: 생초보자 기준)을 임의로 낮추지 않는다.

## 10) 작성 후 점검 체크리스트
- `## 본문`이 존재하는가?
- 세부 주제별 설명이 모두 채워졌는가?
- 예시 코드가 실제 컴파일 가능한가?
- 연습문제와 정답(또는 정답 포인트)이 있는가?
- placeholder/TODO 문구가 남아있지 않은가?
- 상위 문서 링크가 올바른가?
