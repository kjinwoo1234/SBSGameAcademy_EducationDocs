# DocumentReviewSkill

문서 리뷰/개선 업무를 수행할 때 사용하는 AI 가이드라인입니다.  
**핵심 기준은 "비전공자가 처음 학습해도 따라올 수 있는가"** 입니다.

## 1) 목적
- 비전공자/입문자가 막히지 않고 학습을 완료할 수 있는지 평가한다.
- "전공자 관점 정리 품질"보다 "초보자 실행 가능성"을 우선한다.
- 문제점을 찾고, 바로 반영 가능한 문장 수준 개선안을 제시한다.
- 개선 후 예상 점수와 학습 이탈 감소 효과를 함께 제시한다.

## 2) 리뷰 범위
- 전달받은 대상 문서 1개 이상
- 필요 시 연결 문서(README, 다음 학습 문서 링크)까지 점검

## 3) 평가 항목 (100점)
아래 8개 항목을 합산해 100점으로 평가한다.

1. **초보자 진입 난이도** (15점)  
   - 첫 5분 안에 "무엇을 하는 문서인지" 이해 가능한가?
2. **선행지식 의존성 관리** (15점)  
   - 아직 안 배운 개념/용어를 당연하게 쓰지 않는가?
3. **설명 명료성(비전공자 용어)** (15점)  
   - 어려운 용어를 쉬운 말로 풀어 설명했는가?
4. **실습 재현 가능성** (15점)  
   - 문서만 보고 그대로 따라해 성공할 수 있는가?
5. **막힘 방지 장치** (10점)  
   - 자주 하는 실수, 에러 포인트, 점검 항목이 있는가?
6. **학습 흐름 연결성** (10점)  
   - "지금/다음에 무엇을 배우는지" 자연스럽게 이어지는가?
7. **문서 구조/가독성** (10점)  
   - 섹션 순서가 입문자 학습 순서와 맞는가?
8. **예제의 입문 적합성** (10점)  
   - 예제가 짧고 단순하며 바로 실행 가능한가?

### 감점 우선 규칙 (중요)
- 아래 항목이 있으면 총점과 별개로 **보완 필요** 이상 판정:
  1) 선행학습 없는 개념을 설명 없이 사용
  2) 실행 불가능하거나 누락된 예제
  3) 초보자에게 오해를 유발할 수 있는 부정확한 설명

## 4) 리뷰 작성 규칙
- 칭찬보다 문제점/리스크를 우선 제시한다.
- 문제점은 반드시 "왜 문제인지"와 "어떻게 고칠지"를 함께 작성한다.
- 개선안은 추상 표현 금지, 문서에 바로 반영 가능한 작업 단위로 작성한다.
- 점수는 항목별 점수 + 총점 + 감점 근거를 숫자로 제시한다.
- 실제 문서를 직접 읽은 근거를 최소 2개 이상(문장/섹션 단위) 명시한다.
- 리뷰 문장에는 "초보자 기준에서 왜 어려운지"를 반드시 포함한다.
- "전공자에게 익숙하다"는 이유로 감점 항목을 완화하지 않는다.

## 5) 개선안 작성 형식
각 이슈는 아래 형식을 따른다.

- 문제: 현재 상태의 결함
- 영향: 학습자에게 발생하는 혼란/학습 손실
- 수정 제안: 문서에 추가/수정할 구체 내용
- 예상 효과: 개선 후 기대 변화

### 개선안 우선순위 원칙
1. 학습 실패/이탈을 줄이는 수정
2. 실습 성공률을 높이는 수정
3. 문장 다듬기/표현 개선

## 6) 개선 후 재평가
- 제안 반영을 가정한 "개선 후 예상 점수"를 제시한다.
- 상승 폭(예: +14점)과 핵심 상승 요인을 2~3개로 요약한다.
- 개선 후에도 남는 리스크(예: 선행지식 충돌)가 있으면 명시한다.

## 7) 결과물 저장 위치
- 기본 결과 문서는 `./.result/ReviewResult-document.md`로 출력한다.
- 통합 리뷰 수행 시 `./.result/SkillIntegratedReviewResult.md`를 사용한다.

## 8) 출력 템플릿 (필수)
아래 템플릿 구조를 반드시 지킨다.

1. 대상 문서
2. 총평 (적합/보완 필요/부적합)
3. 항목별 점수표 (8개 항목 + 총점)
4. 주요 문제점 TOP 3~5 (심각도 순)
5. 상세 개선안 (즉시 반영 가능한 문장 수준)
6. 개선 후 예상 점수 및 근거
7. 다음 액션 (우선 수정 순서 1~3)

## 9) 초보자 기준 체크 질문 (리뷰 시 필수 자가점검)
리뷰어는 아래 질문에 모두 답하고, 하나라도 "아니오"면 감점 근거로 반영한다.

1. 이 문서는 비전공자가 처음 읽어도 첫 실습을 끝낼 수 있는가?
2. 용어 설명 없이 등장한 개념이 없는가?
3. "왜 하는지"보다 "어떻게 하는지"가 부족하지 않은가?
4. 따라치기 예제가 실제로 컴파일/실행 가능한가?
5. 학습자가 막히는 지점(에러/실수)을 미리 안내하는가?
