# 버전 관리와 협업 기초

개인 개발에서도 버전 관리는 필수이며, 팀 개발에서는 작업 충돌을 줄이는 공통 규칙이 특히 중요합니다.

## 학습 목표
- 커밋/브랜치/병합의 기본 흐름을 설명할 수 있다.
- 작은 단위 커밋으로 작업 이력을 관리할 수 있다.
- 충돌 발생 시 안전하게 해결하고 검증할 수 있다.

## 기본 흐름
1. 작업 전 최신 코드 동기화
2. 기능 단위 브랜치 생성
3. 작은 단위 커밋 작성
4. 병합 전 변경점 리뷰
5. 충돌 해결 후 테스트

## 커밋 메시지 가이드
- 한 줄 요약은 "왜 이 변경이 필요한지" 중심으로 작성
- 예: `입력 처리 버그 수정으로 캐릭터 이동 누락 방지`

## 실수 포인트
- 큰 변경을 한 번에 커밋해 이력 추적이 어려워진다.
- 충돌 해결 후 실행 테스트 없이 바로 병합한다.
- 자동 생성 파일/임시 파일을 그대로 커밋한다.

## 협업 체크리스트
- [ ] 기능 단위 브랜치에서 작업했는가?
- [ ] 커밋 메시지가 변경 목적을 설명하는가?
- [ ] 병합 전후 핵심 기능 테스트를 완료했는가?

## 셀프 퀴즈
1. 왜 "작은 단위 커밋"이 협업에 유리한가?
2. 충돌 해결 직후 반드시 해야 하는 작업은 무엇인가?

## 정답
1. 변경 원인과 영향 범위를 빠르게 추적할 수 있기 때문
2. 실행/테스트로 정상 동작 여부를 검증하는 것
