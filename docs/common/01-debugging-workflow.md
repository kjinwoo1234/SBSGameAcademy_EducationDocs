# 디버깅 워크플로우 기초

코드가 의도대로 동작하지 않을 때는 "추측"보다 "재현 -> 관찰 -> 가설 검증" 순서로 접근해야 빠르게 해결할 수 있습니다.

## 학습 목표
- 버그를 재현 가능한 형태로 정리할 수 있다.
- 로그/브레이크포인트를 사용해 원인을 좁힐 수 있다.
- 수정 후 회귀 확인까지 포함한 디버깅 루프를 수행할 수 있다.

## 기본 흐름
1. **재현**: 언제, 어떤 입력에서 문제 발생하는지 기록
2. **관찰**: 로그 출력/브레이크포인트로 실제 값 확인
3. **가설**: 원인 후보를 1~2개로 축소
4. **검증**: 최소 수정으로 가설을 테스트
5. **회귀 점검**: 관련 기능이 깨지지 않았는지 확인

## 실수 포인트
- 재현 조건을 기록하지 않고 감으로 수정한다.
- 여러 위치를 한 번에 바꿔 원인 추적이 더 어려워진다.
- 버그만 고치고 주변 기능 회귀 테스트를 생략한다.

## 디버깅 체크리스트
- [ ] 재현 절차를 한 줄 이상으로 문서화했는가?
- [ ] 로그 또는 브레이크포인트로 실제 값을 확인했는가?
- [ ] 수정 후 동일 시나리오 + 인접 시나리오를 재검증했는가?

## 셀프 퀴즈
1. 버그 수정 전에 가장 먼저 해야 할 것은 무엇인가?
2. 수정 후 왜 회귀 테스트가 필요한가?

## 정답
1. 버그 재현 조건을 명확히 정의하는 것
2. 수정이 다른 기능에 부작용을 만들 수 있기 때문
