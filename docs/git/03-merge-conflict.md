# Chapter 03 병합 충돌 해결

## 학습 목표
- 병합 충돌이 발생하는 상황을 설명할 수 있다.
- 충돌 마커를 읽고 안전하게 정리할 수 있다.
- 해결 후 검증 루틴을 수행할 수 있다.

## 세부 주제
### 03-1 충돌 발생 원리
- 같은 라인/인접 로직 동시 수정

### 03-2 충돌 마커 해석
- `<<<<<<<`, `=======`, `>>>>>>>`

### 03-3 해결 후 검증
- `git add` + 테스트

## 실습 체크리스트
- 의도적으로 충돌을 만들고 수동 해결까지 완료한다.

## 본문

### 03-1 충돌 발생 원리
- 두 브랜치가 같은 파일의 같은 지점을 다르게 수정하면 자동 병합이 실패할 수 있습니다.
- Git은 어느 쪽이 정답인지 알 수 없어 개발자 판단이 필요합니다.

### 03-2 충돌 마커 해석
- 충돌 파일에는 충돌 범위를 표시하는 마커가 삽입됩니다.
- 각 블록의 의도를 비교한 뒤 최종 코드를 수동으로 확정해야 합니다.

### 03-3 해결 후 검증
- 충돌 부분을 정리한 뒤 `git add`로 해결 완료를 표시합니다.
- 병합은 끝났어도 기능이 깨질 수 있으므로 테스트가 필수입니다.

충돌 마커 예시:
```text
<<<<<<< HEAD
현재 브랜치 코드
=======
병합 대상 브랜치 코드
>>>>>>> feature/xxx
```

용어 설명:
| 용어 | 의미 |
|---|---|
| 충돌(conflict) | 자동 병합이 불가능해 수동 판단이 필요한 상태 |
| 병합 기준(HEAD) | 현재 체크아웃된 브랜치의 최신 커밋 위치 |

예시 명령:
```bash
git status
git add src/player_move.cpp
git commit -m "이동 계산 충돌 해결 후 조건 로직 통합"
```

예상 결과:
```text
충돌 상태가 해제되고 병합 커밋이 완료된다.
```

코드 해석:
- `git status`로 충돌 파일 목록을 먼저 확인합니다.
- `git add`를 해야 Git이 "이 파일 충돌 해결 완료"로 인식합니다.

연습문제:
1. 문제 1 - 충돌 재현
   - 문제: 두 브랜치에서 같은 줄을 다르게 수정해 충돌을 발생시키세요.
   - 입력: 없음
   - 출력: 충돌 마커 포함 파일
   - 조건: 충돌 마커를 수동 정리
2. 문제 2 - 해결 검증
   - 문제: 충돌 해결 후 기능 테스트 체크리스트를 작성/실행하세요.
   - 입력: 테스트 항목 2개 이상
   - 출력: 테스트 통과 결과
   - 힌트: 충돌 지점과 인접 로직 우선 확인

정답 포인트:
- 충돌 해결은 "문자 합치기"가 아니라 "의도 통합"
- 해결 직후 테스트가 실제 품질을 결정
