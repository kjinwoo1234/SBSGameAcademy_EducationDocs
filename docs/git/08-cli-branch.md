# Chapter 8 CLI 환경에서 브랜치 생성 및 조작하기

## 학습 목표

- CLI로 브랜치를 생성·삭제·전환한다.
- 3-way merge 상황을 설명한다.
- 리베이스와 병합의 트레이드오프를 말로 정리한다.
- **태그(tag)**가 브랜치와 어떻게 다른지 말하고, 주석 태그를 만들어 **원격에 공유**하는 흐름을 말한다.

## 세부 주제

- 브랜치 생성(`switch -c` / `branch`)
- 전환·추적 브랜치
- merge와 공통 조상
- rebase 기본
- 태그로 릴리즈 지점(커밋)에 이름 붙이기

## 실습 체크리스트

- 기능 브랜치에서 커밋 2개 → `main`에 merge
- 동일 파일 충돌 1회 해결 후 `merge --continue` 또는 완료 확인
- 연습 저장소에서 **`git tag -a`**로 태그 1개를 만든 뒤 `git tag`로 목록을 확인하고, 원격이 있으면 **`git push origin <태그명>`**까지 시도한다.

## 본문

<a id="ch8-1"></a>
### 8-1 브랜치 생성하기

기능이나 실험을 **기본 브랜치와 분리된 줄기**에서 진행하려면 새 브랜치를 만듭니다. 최근 Git은 **`git switch -c`**로 “만들고 이동”을 한 번에 처리하는 방식이 흔하고, 예전 방식인 **`git branch`로 만든 뒤 `git switch`**로 이동하는 패턴도 동일한 결과를 냅니다.

예시 명령:
```bash
git switch -c feature/cli-demo
# 또는
git branch feature/cli-demo
git switch feature/cli-demo
```

---

<a id="ch8-2"></a>
### 8-2 브랜치 기본 조작하기

`git branch`로 로컬 브랜치 목록을 보고, `git branch -d feature/foo`처럼 **이미 병합된 브랜치**를 정리할 수 있습니다. 원격에만 있는 브랜치를 받아 작업하려면 `git switch -c feature/foo origin/feature/foo`처럼 **추적 브랜치**를 만들면 됩니다.

---

<a id="ch8-3"></a>
### 8-3 3-way 병합하기

**3-way merge**는 **양쪽 브랜치와 공통 조상(common ancestor)** 세 커밋을 기준으로 자동 병합합니다. 충돌이 나면 Git이 세 버전 정보를 바탕으로 파일 안에 **충돌 마커**를 남깁니다.

용어 설명:
| 용어 | 의미 |
|------|------|
| 공통 조상 | 두 브랜치가 갈라지기 직전의 같은 커밋 |

예시 명령:
```bash
git checkout main
git merge feature/cli-demo
```

---

<a id="ch8-4"></a>
### 8-4 리베이스하기

`git rebase main`은 현재 브랜치의 커밋들을 **main 최신 위**로 다시 쌓아 **선형에 가까운 이력**을 만듭니다. 충돌이 나면 `git rebase --continue`로 이어가고, 되돌리려면 `git rebase --abort`로 중단합니다. 이미 팀원과 공유한 브랜치에 무심코 rebase하면 히스토리가 엇갈릴 수 있으므로 팀 규칙을 따릅니다.

---

<a id="ch8-5"></a>

### 8-5 태그로 릴리즈 지점 남기기

**태그(tag)**는 커밋 그래프에서 **특정 커밋 하나를 가리키는 이름 붙은 포인터**입니다. 브랜치가 새 커밋이 생기면 **끝을 따라 이동**하는 것과 달리, 태그는 보통 **출시·빌드·감사 기준**처럼 “이 시점”을 고정해 두려고 씁니다. 예를 들어 `v1.0.0` 같은 **시맨틱 버전** 문자열을 태그 이름으로 쓰면, 나중에 “그 버전의 소스”를 **커밋 해시 대신 이름**으로 가리키기 쉽습니다.

**주석이 있는 태그(annotated tag)**는 `git tag -a`로 메시지·작성자 정보가 들어 있는 태그 객체를 만들고, 릴리즈 노트나 배포 파이프라인과 맞추기에 흔합니다. 이름만 붙이는 **lightweight 태그**는 `git tag v1.0.0`처럼 `-a` 없이 만들 수 있으나, 팀에서는 주석 태그를 권장하는 경우가 많습니다. 원격에 올릴 때는 `git push origin v1.0.0`처럼 **태그 이름을 지정**하거나, 여러 태그를 `git push origin --tags`로 한꺼번에 보내기도 합니다(저장소 규칙·보호 설정에 따름). GitHub **Releases**는 이런 태그에 빌드 산출물·릴리즈 노트를 붙이는 화면으로 이해하면 됩니다([Chapter 3](./03-collaboration-sourcetree.md)의 GUI 맥락과 연결).

예시 명령:

```bash
git tag -a v1.0.0 -m "First stable release"
git tag
git push origin v1.0.0
```

**명령 해석.** `-a`는 주석 태그이며, `git tag`만 실행하면 로컬 태그 목록이 보입니다. 잘못 찍은 태그는 `git tag -d v1.0.0`으로 로컬에서 지울 수 있고, 이미 push했다면 **삭제·갱신은 팀 규칙**에 맞춰야 합니다.

연습문제:

1. 문제: fast-forward merge가 가능한 조건을 한 문장으로 쓰세요.
2. 문제: 팀 규칙이 "main에는 merge commit만"일 때 rebase를 어디(어느 브랜치)에서 주로 쓰게 될지 추론하세요.
3. 문제: **브랜치**와 **태그**가 “가리키는 커밋” 관점에서 어떻게 다르게 움직이는지 한 문장으로 쓰세요.

정답 포인트:

브랜치는 **커밋을 가리키는 포인터**이고, merge는 **3-way 병합**으로 자동 합치다가 충돌할 수 있습니다. rebase는 **커밋을 다시 쌓아 선형 이력**에 가깝게 만들지만, **이미 push한 공유 브랜치**에서는 주의가 필요합니다. 태그는 **한 커밋에 고정된 이름**이고, 일상 개발이 진행돼도 **스스로 따라가지 않는다**는 점에서 브랜치와 구분됩니다.
