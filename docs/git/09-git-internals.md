# Chapter 9 Git 내부 동작 원리

## 학습 목표

- 저장소 루트의 **`.git` 디렉터리**가 하는 일과 대표 하위 항목을 말로 설명한다.
- `git add`가 만드는 객체(블롭·트리) 개념을 말로 설명한다.
- 커밋 객체가 부모·트리·메타데이터를 갖는 이유를 이해한다.
- 브랜치가 refs로 구현된다는 점을 안다.

## 세부 주제

- `.git` 디렉터리 개요(작업 트리와의 관계, 주요 하위 경로)
- 인덱스(스테이징)와 객체 DB
- 커밋 객체 구조
- `git cat-file`로 객체 엿보기
- refs와 HEAD

## 실습 체크리스트

- 저장소 루트에서 `.git` 하위에 `objects`, `refs`, `HEAD`가 있는지 확인한다.
- 로컬 저장소에서 `git cat-file -p HEAD` 실행 후 출력 필드 읽기
- `.git/refs/heads/main` 파일 내용이 커밋 해시인지 확인

## 본문

<a id="ch9-1"></a>

### 9-1 `.git` 디렉터리 개요

**`git init` 또는 `git clone`**을 하면 프로젝트 폴더 안에 **`.git`**이 생깁니다. 여기에 **커밋·브랜치·설정 등 Git이 관리하는 메타데이터**가 모입니다. **작업 디렉터리(working tree)**는 `.git` **밖**에서 편집하는 소스 파일이고, **`.git` 안**은 Git 전용 데이터 영역입니다. 일반 소스는 `.git`에 직접 넣지 않습니다.

| 경로(대표) | 역할 |
|-------------|------|
| `objects/` | 블롭·트리·커밋 등 **객체 DB**(내용은 해시 이름의 파일·팩으로 저장) |
| `refs/heads/` | 로컬 브랜치 끝이 가리키는 **커밋 해시** |
| `refs/remotes/` | 원격 추적 브랜치(예: `origin/main`)의 **커밋 해시** |
| `HEAD` | **지금 체크아웃된 브랜치**(또는 직접 커밋)를 가리키는 참조 |
| `index` | **스테이징 영역**(다음 `git commit`에 넣을 스냅샷 메타데이터) |
| `config` | 이 저장소만의 **설정**(원격 URL, 사용자 이름 등) |
| `hooks/` | 훅 스크립트(커밋 전후 자동 실행 등, 선택) |

**`.git` 폴더를 지우면** 해당 폴더의 **커밋 이력·브랜치 정보가 사라집니다.** 작업 트리의 파일은 남을 수 있어도 Git 관점에서는 새 저장소로 다시 `git init`한 것과 같습니다. 백업 없이 삭제하지 않도록 합니다.

예시 명령(저장소 루트에서 실행, macOS·Linux·Git Bash):

```bash
ls -la .git
```

Windows **PowerShell**에서는 예를 들어 다음과 같이 볼 수 있습니다.

```powershell
Get-ChildItem -Force .git
```

**명령 해석.** `-la`와 `-Force`는 숨김 폴더 `.git`을 목록에 포함하기 위한 옵션입니다. 출력에 `objects`, `refs`, `HEAD` 등이 보이면 이후 절에서 다루는 구조와 연결해 읽으면 됩니다.

연습문제:

1. 문제: 작업 디렉터리와 `.git`의 역할을 한 문장씩으로 구분해 쓰세요.
2. 문제: `index` 파일(또는 스테이징)이 없다면 `git commit` 직전에 어떤 정보를 잃게 될까요? 한 줄로 추론해 보세요.

정답 포인트:

작업 트리는 **편집용 파일**을 두는 곳이고, `.git`은 **버전·참조·설정의 저장소**입니다. 스테이징 없이는 **다음 커밋에 무엇을 넣을지** Git이 일관되게 기록하기 어렵습니다(실제로는 `index`가 그 역할).

### 9-2 git add 명령의 동작 원리

`git add`는 파일 내용을 **해시(blob 객체)**로 `.git/objects`에 저장하고, **인덱스(staging area)**에 “다음 커밋에 넣을 스냅샷”을 기록합니다. 동일 내용은 같은 해시로 **중복 저장을 줄입니다.**

---

<a id="ch9-3"></a>

### 9-3 git commit 명령의 동작 원리

`git commit`은 스테이징된 트리를 가리키는 **tree 객체**와 메타데이터·**부모 커밋 해시**로 **commit 객체**를 만듭니다. 브랜치 포인터(예: `main`)가 새 커밋을 가리키도록 갱신됩니다.

---

<a id="ch9-4"></a>

### 9-4 커밋 객체 살펴보기

예시 명령:

```bash
git cat-file -t HEAD
git cat-file -p HEAD
```

예상 결과(형식 예시):

```text
commit
tree 4b825dc642cb6eb9a060e54bf8d69288fbee4904
parent 7e2c2f5...
author Name <mail> 1710000000 +0900
committer Name <mail> 1710000000 +0900

Commit message here
```

**코드 해석.** `tree` 줄은 프로젝트 디렉터리 스냅샷 루트를 가리키고, `parent`는 직전 커밋입니다(첫 커밋에는 없을 수 있음).

---

<a id="ch9-5"></a>

### 9-5 브랜치 작업 동작 원리

브랜치는 `.git/refs/heads/브랜치명` 파일에 **40자 해시 1줄**을 저장하는 **참조(ref)**입니다. **HEAD**는 보통 `ref: refs/heads/main`처럼 **심볼릭 참조**로 현재 브랜치를 가리킵니다. 체크아웃은 HEAD를 옮기고 작업 트리를 해당 트리로 맞춥니다.

연습문제:

1. 문제: blob, tree, commit 객체 각각이 담는 정보를 한 줄로 정의하세요.
2. 문제: 브랜치를 새로 만들었을 때 `.git` 아래 무엇이 생기거나 바뀌는지 추론하세요.

정답 포인트:

Git은 **콘텐츠 주소 객체 DB**와 **refs**로 이루어져 있고, 커밋은 스냅샷이 아니라 **객체 그래프의 노드**입니다.
