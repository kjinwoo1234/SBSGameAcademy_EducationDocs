# Chapter 10 브랜치 전략은 무엇인가

## 학습 목표

- **브랜치 전략**이 무엇을 약속하는지, **워크플로(workflow, 작업 흐름)**와 어떻게 연결되는지 말로 정의한다.
- 실무에서 자주 이야기되는 **브랜치 관리 전략의 종류**를 이름으로 짚고, 이 과정에서 **어느 챕터**에서 자세히 다루는지 연결한다.
- 팀이나 저장소를 볼 때 **어떤 단서**로 “우리는 어떤 흐름에 가깝다”고 말하기 시작할 수 있는지 한 줄로 말한다.

## 세부 주제

- 브랜치 전략의 정의(규칙 + 협업 흐름)
- 전략이 필요한 이유(동시 작업·추적·배포)
- 대표적인 브랜치 관리 전략의 **종류**(이름과 역할만)
- 이후 챕터에서의 학습 순서 안내

## 실습 체크리스트

- 팀이나 본인이 쓰는 저장소 README·위키에서 **브랜치·PR·릴리즈** 관련 문구를 찾아, 아래 **10-3**에 나온 이름 중 어디에 가장 가까운지 말로 골라 본다.
- [Chapter 8](./08-cli-branch.md)에서 만든 브랜치 이름이, 팀 규칙에서는 보통 **어떤 종류의 브랜치**(예: 기능 전용)로 불릴지 한 줄로 적어 본다.

## 본문

<a id="ch10-1"></a>

### 10-1 브랜치 전략이란

여러 사람이 **같은 Git 저장소**에서 일할 때, **어떤 이름의 브랜치를 언제 만들고 어디에 병합할지**를 팀이 정해 두는 것을 흔히 **브랜치 전략**이라고 부릅니다. `git branch`와 병합·PR(또는 MR) 규칙까지 묶어서, 저장소를 망가뜨리지 않고 협업하는 **흐름 전체**를 말합니다.

전략은 **도구가 강제하는 규칙이 아니라 팀의 약속**입니다. 약속이 없으면 서로 다른 습관으로 같은 브랜치에 덮어쓰거나, 배포에 섞이면 안 될 변경이 한 갈래에 몰리기 쉽습니다.

---

<a id="ch10-2"></a>

### 10-2 왜 규칙이 필요한가

**서로 다른 기능**을 **서로 다른 브랜치**에 두면, 개발 중인 수정이 한꺼번에 섞이지 않고 동시에 진행하기 쉽습니다. 브랜치와 커밋을 **기능·이슈 단위**로 묶어 두면 “이번에 들어간 것”을 추적하고, 필요하면 **그 단위만 되돌리기(rollback)**라고 말하기도 쉬워집니다.

**태그(tag)**는 특정 커밋에 붙는 **이름 붙은 지점**으로, 릴리즈 버전(`v1.0.0` 등)을 표시할 때 자주 씁니다. 브랜치 전략과 맞춰 **어떤 커밋이 곧 배포·감사의 기준인지**를 같이 정하면, 배포 안정성을 이야기할 때 **기준이 팀 안에서 통일**됩니다. CLI에서 태그를 만드는 절차는 [Chapter 8](./08-cli-branch.md) **8-5**를 참고합니다.

---

<a id="ch10-3"></a>

### 10-3 흔히 쓰는 브랜치 관리 전략의 종류

같은 Git이라도 팀마다 **브랜치를 몇 개 두고, 어떤 이름으로 나누는지**가 다릅니다. 교육 과정에서는 아래처럼 **이름이 붙은 대표 패턴**을 나누어 뒤이어 장에서 풀어 갑니다. 여기서는 **종류와 챕터 연결**만 기억하면 됩니다.

| 전략 이름 | 한 줄 위치 잡기 | 자세한 설명 |
|-----------|-----------------|-------------|
| **GitHub Flow** | 기본 브랜치 하나를 축으로 짧은 기능 브랜치·PR을 돌리는 **단순한 흐름**에 가깝다. | [Chapter 11](./11-github-flow.md) |
| **GitLab Flow** | GitHub Flow에 가깝게 가되, **스테이징·운영** 같은 **환경**을 브랜치 이름 등으로 드러내는 팀이 많다. | [Chapter 12](./12-gitlab-flow.md) |
| **Git Flow** | `main`, `develop`, `feature`, `release`, `hotfix`처럼 **역할별 브랜치**를 두고 릴리즈 절차를 나누는 **전통적인 모델**이다. | [Chapter 13](./13-git-flow.md) |

각 패턴의 **흐름 도식**은 위 표의 챕터 본문(11·12·13)에서 해당 설명과 함께 둡니다. 세부 절차·실습도 같은 장에서 이어집니다.

그 밖에도 **팀만의 규칙**(접두어, 보호 브랜치, 릴리즈 담당)을 얹거나, 위 이름을 **섞어 쓰는 하이브리드**가 흔합니다. 여러 전략을 **표로 비교**하고 **Fork·`origin`·`upstream`**까지 묶어 보려면 [Chapter 14](./14-branch-comparison-fork-team-pr.md)로 이어가면 됩니다.

<a id="ch10-4"></a>

### 10-4 이후 챕터로 이어 읽기

이 장은 **용어와 지도**만 맞춘 것입니다. **GitHub Flow**를 단계별로 실습하려면 [Chapter 11](./11-github-flow.md)으로, **환경을 브랜치로 표현하는 흐름**은 [Chapter 12](./12-gitlab-flow.md)으로, **역할별 브랜치가 많은 Git Flow**는 [Chapter 13](./13-git-flow.md)으로 넘어가면 됩니다. 세 가지를 한눈에 비교하고 협업·포크까지 묶으면 [Chapter 14](./14-branch-comparison-fork-team-pr.md)입니다.

연습문제:

1. 문제: “브랜치 전략”을 **한 문장**으로 정의하세요(도구 이름만 나열하지 말 것).
2. 문제: **10-3** 표에 나온 세 가지 이름 중, “브랜치 **역할 종류**가 가장 많이 나뉜다고 알려진 패턴”은 무엇인지 한 단어로 쓰세요.
3. 문제: 저장소 README에 전략 이름이 없을 때, **어떤 단서**(예: 브랜치 이름 규칙, PR 규칙, 배포 문구)를 보면 팀 흐름을 추측하기 쉬운지 한 가지 이상 적으세요.

정답 포인트:

브랜치 전략은 **협업을 위해 브랜치 생명주기·이름·병합 순서를 팀이 정한 약속**(워크플로 포함)으로 말할 수 있습니다. 역할별 브랜치가 많이 나뉜 패턴으로 널리 알려진 이름은 **Git Flow**입니다. README가 없어도 **기본 브랜치 이름**, **보호 규칙·PR 템플릿**, **배포·환경 관련 브랜치** 같은 단서로 실제 운영에 가까운 흐름을 짐작할 수 있습니다.
