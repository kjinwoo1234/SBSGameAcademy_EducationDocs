# 브랜치와 커밋 워크플로우

기능 단위 브랜치 전략은 충돌을 줄이고 리뷰 품질을 높입니다.

## 학습 목표
- 기능 브랜치를 생성하고 전환할 수 있다.
- 작은 단위 커밋 기준을 적용할 수 있다.
- 커밋 메시지를 "왜" 중심으로 작성할 수 있다.

## 권장 흐름
1. `main` 최신화
2. 기능 브랜치 생성
3. 기능 단위 커밋
4. 브랜치 정리 및 리뷰 요청

## 핵심 명령어
```bash
git switch main
git pull origin main
git switch -c feature/input-validation

git add src/input.c
git commit -m "잘못된 숫자 입력 처리로 런타임 오류 감소"
```

## 단계별 실습
1. `main` 브랜치에서 `git pull origin main`으로 최신 상태를 맞춘다.
2. `git switch -c feature/input-validation`으로 기능 브랜치를 만든다.
3. 입력 검증 코드를 추가한 뒤 `git status`로 변경 파일을 확인한다.
4. 기능 단위로 `git add`와 `git commit`을 수행한다.
5. 커밋 메시지를 읽고 "왜 변경했는지"가 드러나는지 점검한다.

## 좋은 커밋 기준
- 하나의 커밋은 하나의 의도만 담는다.
- 코드 변경 + 관련 문서 변경을 함께 묶는다.
- 테스트 가능한 상태로 커밋한다.

## 실수 포인트
- 서로 관련 없는 수정들을 한 커밋에 섞는 문제
- 브랜치 이름이 목적을 설명하지 못하는 문제

## 이해 점검 체크리스트
- 브랜치 생성 전에 기준 브랜치를 최신화했는가?
- 한 커밋에 하나의 변경 목적만 포함했는가?
- 커밋 메시지에 변경 이유가 드러나는가?

## 셀프 퀴즈
1. 기능 브랜치 이름에 `feature/`, `fix/` 접두어를 쓰는 이유는?
2. 큰 기능을 여러 커밋으로 쪼개면 어떤 이점이 있는가?

## 정답
1. 작업 성격을 빠르게 식별하고 협업 시 의사소통 비용을 줄이기 위해서
2. 리뷰/디버깅/롤백이 쉬워지고 변경 영향 추적이 정확해진다.
