# Chapter 02 브랜치와 커밋 워크플로우

## 학습 목표
- 기능 브랜치를 생성/전환하고 작업을 분리할 수 있다.
- 하나의 의도만 담는 커밋 습관을 적용할 수 있다.
- 변경 이유 중심의 커밋 메시지를 작성할 수 있다.

## 세부 주제
### 02-1 브랜치 전략
- `main` 보호, 기능 브랜치 작업

### 02-2 기능 단위 커밋
- 작은 변경 묶음

### 02-3 커밋 메시지 품질
- "무엇"보다 "왜"

## 실습 체크리스트
- 기능 브랜치를 만들고 2개 이상의 기능 단위 커밋을 작성한다.

## 본문

### 02-1 브랜치 전략
- 협업에서는 `main`을 안정 상태로 유지하고 기능 작업은 별도 브랜치에서 수행합니다.
- 브랜치 분리로 충돌 범위를 줄이고 코드 리뷰를 쉽게 만듭니다.

### 02-2 기능 단위 커밋
- 하나의 커밋에는 하나의 목적만 담는 것이 좋습니다.
- 되돌리기/원인 추적/리뷰가 쉬워집니다.

### 02-3 커밋 메시지 품질
- 메시지는 변경 이유를 드러내야 팀원이 의도를 빠르게 이해할 수 있습니다.
- "수정", "업데이트" 같은 모호한 문구는 피합니다.

브랜치 이름 예시:
| 작업 성격 | 브랜치 이름 예시 |
|---|---|
| 기능 추가 | `feature/input-validation` |
| 버그 수정 | `fix/login-null-check` |
| 문서 보완 | `docs/git-workflow-update` |

용어 설명:
| 용어 | 의미 |
|---|---|
| 기준 브랜치(base branch) | 기능 브랜치를 만들 때 기준이 되는 안정 브랜치(보통 `main`) |
| 원자 커밋(atomic commit) | 하나의 변경 의도만 담은 작고 독립적인 커밋 |

예시 명령:
```bash
git switch main
git pull origin main
git switch -c feature/input-validation
git add src/input.c
git commit -m "숫자 입력 검증 추가로 잘못된 입력 시 종료 문제 방지"
```

예상 결과:
```text
main과 분리된 기능 브랜치에서 의도 단위 커밋이 누적된다.
```

코드 해석:
- 브랜치를 먼저 분리한 뒤 커밋하면 변경 영향 범위를 명확히 유지할 수 있습니다.
- 커밋 메시지에 목적을 쓰면 리뷰 시 판단 속도가 빨라집니다.

연습문제:
1. 문제 1 - 기능 브랜치 생성
   - 문제: `feature/ui-color` 브랜치를 만들고 파일 1개를 수정해 커밋하세요.
   - 입력: 없음
   - 출력: 브랜치 1개, 커밋 1개
   - 조건: 커밋 메시지에 변경 이유 포함
2. 문제 2 - 커밋 분리
   - 문제: 기능 변경과 문서 변경을 서로 다른 커밋으로 분리하세요.
   - 입력: 코드 수정 + 문서 수정
   - 출력: 커밋 2개
   - 힌트: `git add <파일>` 선택 스테이징 사용

정답 포인트:
- 브랜치 분리 + 커밋 분리가 협업 품질 핵심
- 메시지 품질은 코드 품질만큼 중요
