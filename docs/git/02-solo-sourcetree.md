# Chapter 2 혼자서 Git으로 버전 관리하기

## 학습 목표
- SourceTree로 기존 로컬 저장소를 열거나 새로 만든다.
- 스테이징·커밋·푸시를 GUI 흐름으로 수행한다.
- 커밋 그래프(히스토리)를 읽을 수 있다.

## 세부 주제
- 로컬 저장소를 SourceTree에 추가
- 커밋과 푸시
- 커밋 그래프로 이력 이해하기

## 실습 체크리스트
- SourceTree에서 `File > Add`로 로컬 Git 폴더 등록
- 커밋 2개 이상 후 원격에 푸시
- 로그/그래프 뷰에서 부모-자식 커밋 관계 확인

## 본문

<a id="ch2-1"></a>
### 2-1 로컬 저장소를 소스트리에 불러오기

- **Add** 또는 **Clone**을 구분합니다. 이미 `git init`한 폴더는 **Add(로컬 저장소 추가)**입니다.
- 저장소 경로를 지정하면 브랜치 목록과 파일 상태가 표시됩니다.
- 원격이 없다면 나중에 **Repository settings**에서 `origin` URL을 추가합니다.

연습문제:
1. 문제: `git init`만 한 폴더와 `git clone`으로 받은 폴더를 SourceTree에 각각 추가해 보고, 원격 표시 차이를 적으세요.

정답 포인트:
- 클론은 보통 원격이 미리 잡혀 있고, 로컬만 init한 경우 원격을 수동 추가.

---

<a id="ch2-2"></a>
### 2-2 소스트리로 커밋 만들고 푸시하기

- 변경 파일을 **스테이징**(Stage)한 뒤 커밋 메시지를 작성하고 **Commit**합니다.
- 푸시할 브랜치를 선택하고 **Push**합니다. 첫 푸시는 upstream 설정이 필요할 수 있습니다.
- 커밋 메시지는 **무엇을·왜** 바꿨는지 한 줄 요약 + 필요 시 본문을 권장합니다.

예시 명령(동일 작업을 CLI로 했을 때):
```bash
git add .
git commit -m "Describe change"
git push -u origin main
```

코드 해석:
- GUI의 Stage/Commit/Push는 위 CLI와 1:1로 대응합니다.

연습문제:
1. 문제: 의도적으로 일부 파일만 스테이징하고 커밋한 뒤, 나머지 파일은 다음 커밋으로 나누세요.

정답 포인트:
- 원자 커밋(한 커밋에 한 논리적 변경) 습관이 리뷰와 되돌리기에 유리합니다.

---

<a id="ch2-3"></a>
### 2-3 그림으로 Git 뜯어보기

- **커밋 그래프**는 점(커밋)과 선(부모 관계)으로 시간 순 이력을 보여 줍니다.
- **브랜치 레이블**(`main`, `feature/...`)은 특정 커밋을 가리키는 **움직이는 포인터**입니다.
- **HEAD**는 "지금 작업 중인 위치"를 가리킵니다(보통 브랜치 이름을 통해).

용어 설명:
| 용어 | 의미 |
|------|------|
| HEAD | 현재 체크아웃된 커밋(또는 그 브랜치)을 가리키는 포인터 |

연습문제:
1. 문제: 새 브랜치를 만든 뒤 커밋 1개를 추가하고, 그래프에서 `main`과 새 브랜치가 갈라진 지점을 찾아 스크린샷 또는 설명으로 남기세요.

정답 포인트:
- 브랜치는 "이름 붙은 커밋 포인터"; 병합 전까지는 그래프가 갈라져 보입니다.
