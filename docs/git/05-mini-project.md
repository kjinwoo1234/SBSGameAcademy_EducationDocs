# Chapter 05 Git 미니 프로젝트: 팀 협업 시뮬레이션

## 학습 목표
- 브랜치 생성부터 병합까지 협업 흐름을 끝까지 재현할 수 있다.
- 충돌 발생 시 해결과 검증을 체계적으로 수행할 수 있다.
- PR 문서화 품질(이유/검증)을 갖춘 제출물을 만들 수 있다.

## 세부 주제
### 05-1 시나리오 기반 브랜치 전략
- 기능 A/B 분리

### 05-2 충돌 해결 실습
- 병합 순서와 수동 통합

### 05-3 PR 품질 점검
- 변경 이유/테스트 결과 정리

## 실습 체크리스트
- 기능 2개를 각 브랜치에서 작업하고 최종 병합까지 완료한다.

## 본문

### 05-1 시나리오 기반 브랜치 전략
- 팀 협업 시뮬레이션에서는 기능을 브랜치별로 분리해 병렬 개발을 연습합니다.
- 기능 단위 분리는 충돌 범위를 줄이고 리뷰 품질을 높입니다.

### 05-2 충돌 해결 실습
- 병합 순서에 따라 충돌 지점이 달라질 수 있어 `status` 확인이 중요합니다.
- 충돌 해결 후에는 즉시 테스트를 수행해 논리 누락을 확인해야 합니다.

### 05-3 PR 품질 점검
- PR에는 "무엇을 바꿨는지"보다 "왜 바꿨는지"가 먼저 드러나야 합니다.
- 테스트 항목 체크리스트를 함께 제출하면 리뷰 신뢰도가 올라갑니다.

프로젝트 시나리오:
| 항목 | 내용 |
|---|---|
| 기능 A | 플레이어 이동 입력 보정 |
| 기능 B | UI 점수 표시 개선 |
| 목표 | 기능 A/B를 병합해 `main` 안정 상태 유지 |

용어 설명:
| 용어 | 의미 |
|---|---|
| 시뮬레이션(simulation) | 실제 협업 상황을 축소해 단계적으로 재현하는 학습 방식 |
| 검증 근거 | 테스트 결과/로그처럼 변경이 안전함을 증명하는 자료 |

예시 명령:
```bash
git switch main
git pull origin main
git switch -c feature/player-input-fix
git add .
git commit -m "입력 보정 조건 추가로 이동 누락 감소"
git push -u origin feature/player-input-fix
```

예상 결과:
```text
기능 브랜치 2개, PR 2개, 최종 main 병합 로그가 남는다.
```

코드 해석:
- 브랜치 분리를 먼저 수행하면 기능 간 영향 범위를 독립적으로 관리할 수 있습니다.
- 푸시 후 PR로 리뷰를 거치면 병합 전 품질 점검이 가능합니다.

연습문제:
1. 문제 1 - 1인 2역 협업 시뮬레이션
   - 문제: 기능 A/B를 각각 별도 브랜치에서 구현하고 병합하세요.
   - 입력: 기능 요구사항 2개
   - 출력: 병합 완료 로그
   - 조건: 커밋 메시지에 변경 이유 포함
2. 문제 2 - 충돌 해결 보고서
   - 문제: 충돌 발생 지점, 해결 방법, 테스트 결과를 3줄 이상 기록하세요.
   - 입력: 충돌 사례 1건
   - 출력: 해결 보고
   - 힌트: 충돌 전/후 코드 의도 비교

정답 포인트:
- 브랜치 전략, 충돌 해결, PR 품질은 하나의 흐름으로 연결됨
- 병합 결과만이 아니라 "검증 근거"까지 남겨야 협업 품질이 올라감
