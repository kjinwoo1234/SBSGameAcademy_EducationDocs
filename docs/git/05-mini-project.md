# Git 미니 프로젝트: 팀 협업 시뮬레이션

이 문서는 Git 기초부터 원격 협업까지 한 번에 연결해보는 통합 실습입니다.

## 학습 목표
- 기능 브랜치 기반 협업 흐름을 처음부터 끝까지 재현할 수 있다.
- 충돌 발생 시 안전하게 해결하고 검증할 수 있다.
- Pull Request 기준으로 변경 이유와 테스트 결과를 정리할 수 있다.

## 프로젝트 시나리오
- 팀 구성: 2명(또는 1인 2역)
- 기능 A: 플레이어 이동 입력 보정
- 기능 B: UI 점수 표시 개선
- 목표: 두 기능을 각각 브랜치에서 작업 후 메인 브랜치에 병합

## 단계별 실습
1. 기준 브랜치 최신화: `git switch main` -> `git pull origin main`
2. 기능 A 브랜치 생성 후 작업/커밋: `feature/player-input-fix`
3. 기능 B 브랜치 생성 후 작업/커밋: `feature/ui-score-update`
4. 두 브랜치를 순서대로 병합하며 충돌이 발생하면 수동 해결
5. PR 본문에 변경 목적/테스트 결과를 작성하고 병합 완료
6. 병합 후 `main` 재동기화 및 사용 브랜치 정리

## 필수 명령어 예시
```bash
git switch main
git pull origin main
git switch -c feature/player-input-fix
git add .
git commit -m "입력 보정 조건 추가로 이동 누락 감소"
git push -u origin feature/player-input-fix
```

## 검증 기준
- 브랜치별 커밋 목적이 명확히 분리되어 있는가?
- 충돌 해결 후 기능 A/B가 모두 정상 동작하는가?
- PR 설명에 변경 이유와 테스트 결과가 포함되어 있는가?

## 제출 산출물
1. 브랜치 목록 캡처 또는 로그
2. PR 설명 본문(요약 + 테스트 결과)
3. 병합 후 `git log --oneline` 결과

## 셀프 퀴즈
1. 기능 단위 브랜치가 충돌 관리에 유리한 이유는?
2. 충돌 해결 후 즉시 병합하지 않고 테스트해야 하는 이유는?

## 정답
1. 변경 범위를 분리해 영향 추적과 리뷰를 쉽게 만들기 때문
2. 수동 병합 과정에서 논리 오류/누락이 발생할 수 있기 때문
