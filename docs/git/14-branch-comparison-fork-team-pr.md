### 14-2 Fork가 있는 이유, origin과 upstream

**Fork(포크)**는 원본 저장소를 **내 계정 아래에 복제한 별도 원격**입니다. 원본에 **직접 푸시 권한이 없어도** 실험·커밋·브랜치를 만들 수 있어 **오픈소스 기여**에 자주 쓰입니다. 로컬에서 클론한 저장소는 보통 **`origin`**이 **내가 푸시하는 원격**(대개 내 포크)을 가리킵니다. 원본을 따로 등록할 때 관습적으로 **`upstream`**이라는 이름을 씁니다. **둘 다 원격 URL에 붙인 별칭**이며, 팀마다 이름을 다르게 줄 수도 있습니다.

| 별칭 | 흔한 가리키는 대상 |
|------|---------------------|
| `origin` | 내 포크(또는 팀이 나에게 준 쓰기 가능 원격) |
| `upstream` | 포크의 원본(조직·메인테이너 저장소) |

**포크를 쓰지 않을 때의 한계**는 다음과 같습니다. 같은 원격에 쓰기 권한을 모두에게 주면 **브랜치 이름 충돌·실수 푸시**가 늘 수 있고, 오픈소스에서는 **검증되지 않은 기여자에게 원본 쓰기 권한을 줄 수 없으므로** 포크 + PR이 표준이 됩니다. `git remote add`·`fetch`·`merge`/`rebase`로 맞추는 **구체 명령**은 [Chapter 4 둘 이상의 원격 저장소로 협업하기](./04-multi-remote.md)에서 다룹니다.

### 14-3 팀 전략과 개인 Fork·PR을 함께 쓰기

**프로젝트가 커지면** 브랜치 이름만으로 모든 기여를 정리하기 어렵고, 원본 저장소의 **권한·감사** 요구가 커집니다. 조직 저장소에서는 GitHub Flow 등 **팀 공통 규칙**과 **`main` 보호**([Chapter 6](./06-github-features.md))를 두는 경우가 많습니다. **외부 기여자**는 원본에 브랜치를 만들 권한이 없거나, **포크에서 작업**하는 편이 **권한 분리·실수 격리**에 유리합니다. 개인은 **자기 포크**에서 브랜치를 쌓고, 준비가 되면 **원본으로 PR**을 열어 팀 규칙 안으로 들어옵니다.

**오류 방지 패턴(혼동 줄이기)**을 정리하면 다음과 같습니다. **언제** 문제가 잘 나오냐 하면, 원본 `main`에 직접 푸시할 권한이 없거나 팀이 포크 기반 기여를 요구할 때입니다. **왜** 곤란한지는, 포크의 `main`만 맞추고 **upstream 최신화를 빼먹으면** PR에 **옛날 코드 기준 패치**가 섞이거나 충돌이 커지기 때문입니다. **어떻게** 막느냐면, 작업 전 **`upstream`에서 가져와** 브랜치 베이스를 맞춘 뒤 PR을 갱신합니다(리베이스 여부는 팀 규칙에 따름).

연습문제:

1. 문제: 소규모 웹 서비스 팀이 **매일 배포**한다면 Git Flow 전체를 그대로 도입하기보다 무엇부터 익히는 것이 보통 부담이 적은지 한 문장으로 설명하세요.
2. 문제: `origin`과 `upstream`을 각각 한 문장으로 정의하세요(누구 저장소인지 중심으로).
3. 문제: “팀은 GitHub Flow로 `main`을 관리하고, 기여자는 포크에서만 작업한다”일 때, 기여자가 **원본 저장소에 직접 브랜치를 만들 필요가 없는** 이유를 적으세요.

정답 포인트:

배포가 잦으면 **GitHub Flow + 보호 브랜치**가 먼저 맞는 경우가 많고, Git Flow는 **릴리즈·역할 분리**가 분명할 때 가치가 큽니다. `origin`은 내가 푸시하는 쪽(대개 포크), `upstream`은 합쳐지길 원하는 원본으로 이해하면 됩니다. 포크는 **권한·실수 범위**를 나누고, PR로 **팀 규칙 안으로만** 변경이 들어오게 합니다.
