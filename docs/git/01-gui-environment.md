# Chapter 1 GUI를 위한 버전 관리 환경 구축하기

## 학습 목표
- SourceTree와 Visual Studio Code를 설치하고 기본 설정을 마친다.
- GitHub 웹 UI에서 저장소·이슈·PR 화면의 역할을 구분한다.

## 세부 주제
- SourceTree 설치와 Git 연동
- Visual Studio Code 설치와 Git 확장 활용
- GitHub 저장소 화면 탐색

## 실습 체크리스트
- SourceTree에서 Git 실행 파일 경로가 인식되는지 확인
- VS Code에서 폴더를 열고 소스 제어 탭을 연다
- GitHub에서 임의 공개 저장소의 Code / Issues / Pull requests 탭을 연다

## 본문

<a id="ch1-1"></a>
### 1-1 소스트리 설치하기

- **SourceTree**는 Atlassian에서 제공하는 **Git GUI 클라이언트**입니다. 명령어 대신 클릭으로 커밋·푸시·브랜치를 다룰 수 있습니다.
- 설치 과정에서 **내장 Git 사용** 또는 **시스템 Git** 중 하나를 선택합니다. 이미 Git for Windows를 썼다면 시스템 Git을 지정해도 됩니다.
- 첫 실행 시 Atlassian 계정 연동을 요구할 수 있습니다. 교육용으로 계정 없이 진행 가능한 버전 정책은 공식 안내를 따릅니다.

용어 설명:
| 용어 | 의미 |
|------|------|
| GUI | Graphical User Interface, 마우스·화면 중심 인터페이스 |

연습문제:
1. 문제: SourceTree의 "파일 상태" 화면에서 스테이징 영역이 무엇을 의미하는지 자신의 말로 설명하세요.

정답 포인트:
- GUI는 Git의 동일한 개념(작업 트리 → 스테이징 → 커밋)을 시각화합니다.

---

<a id="ch1-2"></a>
### 1-2 비주얼 스튜디오 코드 설치하기

- **Visual Studio Code(VS Code)**는 편집기이며, **내장 Git 뷰**로 diff 확인·스테이징·커밋·푸시를 할 수 있습니다.
- 확장 마켓에서 **GitHub Pull Requests** 등을 추가하면 PR 검토가 편해집니다.
- 터미널을 VS Code 안에서 열면 CLI와 GUI를 섞어 쓰기 좋습니다.

연습문제:
1. 문제: VS Code에서 폴더를 연 뒤 소스 제어 아이콘에서 변경 파일 1개를 스테이징하고 커밋 메시지를 작성해 보세요(실제 커밋까지).

정답 포인트:
- 편집·검색·Git이 한 화면에 모이면 작업 속도가 안정됩니다.

---

<a id="ch1-3"></a>
### 1-3 GitHub 둘러보기

- **Code** 탭: 파일 트리, 브랜치 전환, 최근 커밋 요약.
- **Issues** 탭: 할 일·버그·토론을 이슈로 관리.
- **Pull requests** 탭: 브랜치 병합 전 **코드 리뷰**와 대화가 이뤄집니다.

약어 설명(PR):
- **PR**은 **Pull Request**의 약자입니다.
- 한국어로는 "풀 리퀘스트(병합 요청)"라고 부르며, 내 브랜치의 변경을 원본에 합쳐 달라고 요청하는 협업 절차입니다.
- 리뷰·CI 통과·승인 후 병합하는 흐름을 표준화하는 데 쓰입니다.

| 탭 | 주 용도 |
|----|---------|
| Actions | CI/CD 워크플로(자동 빌드·테스트) |
| Settings | 브랜치 보호, 시크릿, 협업자 |

연습문제:
1. 문제: 임의 오픈소스 저장소에서 열려 있는 PR 1개를 열고, "대화(Conversation)"와 "파일 변경(Files changed)"의 차이를 설명하세요.

정답 포인트:
- PR은 **변경 제안 + 리뷰 + 병합**까지를 한 곳에서 다루는 GitHub의 협업 단위입니다.
