# Chapter 26 테스트/디버깅 기초

## 학습 목표
- 재현 → 관찰 → 가설 → 검증 순서로 버그를 추적할 수 있다.
- 중단점과 핵심 로그를 구분해 상태를 확인할 수 있다.
- `Debug.Assert`와 `try-catch`로 정상·예외 케이스를 검증할 수 있다.

## 본문

### 26-1 디버깅 기본 흐름

디버깅은 감으로 고치는 일이 아니라 **절차**입니다. 같은 입력으로 **재현**하고, 변수·흐름을 **관찰**하고, 원인 **가설**을 하나 고른 뒤, 수정 후 같은 케이스로 **검증**합니다.

1. 재현: 같은 입력에서 같은 오류가 나는지 확인
2. 관찰: 변수 값과 분기 결과를 수집
3. 가설: 원인 후보를 하나로 좁힘
4. 검증: 수정 후 재현 케이스를 다시 실행

아래는 관찰용 로그가 있는 예입니다. 실행해 `b`가 `0`일 때 어디서 멈추는지 읽어 보세요. `Divide` 줄은 주석으로 두었습니다. 켜면 예외가 납니다.

```csharp
using System;

class Program
{
    static int Divide(int a, int b)
    {
        Console.WriteLine($"입력: a={a}, b={b}");
        if (b == 0)
        {
            Console.WriteLine("분기: b가 0");
            throw new ArgumentException("b는 0일 수 없습니다.");
        }
        int result = a / b;
        Console.WriteLine($"반환값: {result}");
        return result;
    }

    static void Main()
    {
        try
        {
            Divide(10, 0);
        }
        catch (ArgumentException ex)
        {
            Console.WriteLine(ex.Message);
        }
    }
}
```

**예상 출력**

```text
입력: a=10, b=0
분기: b가 0
b는 0일 수 없습니다.
```

**코드 해석**
- 입력 로그로 재현에 쓰는 값을 남깁니다.
- 분기 로그로 `b == 0` 경로에 들어갔음을 확인합니다.
- 예외 메시지로 “0으로 나눔을 막았는지”를 검증합니다.

### 26-2 중단점과 로그

**중단점**은 특정 줄에서 실행을 잠시 멈춰, 그 순간 변수 값을 보는 기능입니다. Visual Studio나 Cursor 같은 편집기에서 줄 번호 왼쪽을 클릭하면 빨간 점으로 표시됩니다. **조사식**은 멈춰 있는 동안 관심 변수나 식을 계속 보여 주는 창입니다. **로그**는 `Console.WriteLine`으로 흐름을 출력하는 방법입니다. 빠르지만 줄이 너무 많으면 오히려 길을 잃기 쉽습니다. “입력 / 분기 결과 / 반환값”처럼 핵심 몇 줄만 남기는 편이 낫습니다.

중단점으로 상태를 볼 때는 대략 다음 순서를 따릅니다.

1. 의심되는 줄 왼쪽에 중단점을 켠다
2. 디버그 실행으로 프로그램을 시작한다
3. 멈춘 줄에서 `a`, `b` 같은 변수 값을 확인한다
4. 한 줄씩 진행하며 분기와 반환을 따라간다

| 방법 | 적합한 상황 | 장점 | 주의점 |
|---|---|---|---|
| 중단점·조사식 | 한 순간 상태를 자세히 볼 때 | 값을 직접 확인 | 실행을 멈춰야 함 |
| `Console.WriteLine` 로그 | 빠른 추적, 콘솔만 있을 때 | 설정이 단순 | 로그가 많아지면 읽기 어려움 |

### 26-3 단위 테스트 기초

**단위 테스트**는 작은 기능을 자동으로 확인해 **회귀**를 줄입니다. 회귀는 수정 이후 기존 기능이 다시 깨지는 문제입니다. 최소로 **정상 입력**, **경계값**, **예외 상황**을 하나씩 둡니다. 입문에서는 `Debug.Assert`와 `try-catch`로도 연습할 수 있습니다. `Debug.Assert(조건);`은 조건이 `false`이면 Debug 구성에서 실패를 드러냅니다.

| 용어 | 의미 |
|---|---|
| 회귀 | 수정 이후 기존 기능이 다시 깨지는 문제 |
| 단위 테스트 | 함수·메서드 단위의 자동 검증 |
| `Debug.Assert` | 조건이 참인지 확인하고, 아니면 디버그에서 실패를 표시 |

아래를 실행해 Assert가 통과하고 `예외 테스트 통과`가 출력되는지 확인해 보세요.

```csharp
using System;
using System.Diagnostics;

class Calculator
{
    public static int DivideSafe(int a, int b)
    {
        if (b == 0)
        {
            throw new ArgumentException("b는 0일 수 없습니다.");
        }
        return a / b;
    }
}

class Program
{
    static void Main()
    {
        Debug.Assert(Calculator.DivideSafe(10, 2) == 5);
        Debug.Assert(Calculator.DivideSafe(-9, 3) == -3);

        try
        {
            Calculator.DivideSafe(1, 0);
        }
        catch (ArgumentException)
        {
            Console.WriteLine("예외 테스트 통과");
        }
    }
}
```

**예상 출력**

```text
예외 테스트 통과
```

**코드 해석**
- `Debug.Assert(... == 5)`로 정상 나눗셈 결과를 확인합니다.
- 음수 나눗셈도 Assert로 한 번 더 확인합니다.
- `b == 0`일 때 예외가 나는지는 `try-catch`로 검증합니다.

이 장에서는 디버깅 절차와 최소 검증까지입니다. xUnit 같은 테스트 프레임워크 설치는 다루지 않습니다.

### 연습문제

등급 규칙: `90` 이상 `A`, `60` 이상 `D`, 그 미만 `F`. 이 장 연습에서는 `B` 등급은 쓰지 않습니다.

**문제 1**
- 문제: 점수→등급 함수를 만들고 아래 점수에 대해 Assert 또는 출력 비교로 검증하세요.
- 입력: 점수 `59`, `60`, `90`, `100`
- 출력:
  ```text
  F
  D
  A
  A
  ```
- 조건: 위 등급 규칙을 따를 것. 경계값 `59`·`60`·`90` 포함

**문제 2**
- 문제: 로그가 많은 함수에서 핵심 로그 3개만 남기세요.
- 입력: 없음. `26-1`의 `Divide`처럼 입력·분기·반환을 찍는 함수를 기준
- 출력: 남긴 로그 역할이 아래와 같으면 통과
  ```text
  입력
  분기 결과
  반환값
  ```
- 조건: 세 역할만 남기고, 각 역할이 왜 필요한지 한 줄씩 적기

**문제 3**
- 문제: `DivideSafe`에서 `b == 0` 검사가 빠진 버그를 가정하고, 재현·검증 체크리스트를 작성하세요.
- 입력: 없음. 예 입력 `a=1`, `b=0`
- 출력: 아래 형식으로 채운 기록
  - 재현: 입력 / 기대 / 실제
  - 수정: 한 줄
  - 검증: 다시 실행한 결과
- 조건: 재현 가능하게 숫자·메시지를 명시

### 정답 포인트

- 문제 1: `59→F`, `60→D`, `90→A`, `100→A`를 Assert 또는 출력으로 확인
- 문제 2: 입력·분기·반환 로그만 남기고 나머지 삭제
- 문제 3: 같은 입력으로 수정 전후를 비교. 예) 기대는 예외 메시지, 실제는 0 나눗셈 예외 등
- 공통: 증거 중심 디버깅, 테스트는 최소 안전망
