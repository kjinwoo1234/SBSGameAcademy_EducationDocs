# Chapter 09 GAS(Gameplay Ability System)

## 학습 목표
- ASC, Ability, Attribute, Effect, Tag, Cue의 역할을 한 줄씩 정의할 수 있다.
- 스킬·버프·디버프를 GAS 관점에서 설계할 수 있다.

## 세부 주제
- ASC, 게임플레이 어빌리티, 어트리뷰트, 이펙트, 태그, 큐

## 실습 체크리스트
- 프로젝트에 **Gameplay Abilities** 플러그인을 활성화하고 **Ability System Component**를 Pawn에 부착한다.
- 샘플 **Attribute Set**(Health, MaxHealth)과 **Gameplay Effect**로 회복·피해를 적용한다.

## 본문

### 9-1 ASC(Ability System Component)

약어 설명(GAS):
- **GAS**는 **Gameplay Ability System**의 약자입니다.
- 한국어로는 "게임플레이 어빌리티 시스템"이라 부르며, 스킬·스탯·버프·태그를 **일관된 규칙**으로 묶는 프레임워크입니다.
- 단순 `Apply Damage`만 쓰는 구조보다 규모 있는 전투·멀티플레이에서 **확장과 디버깅**을 쉽게 하기 위해 도입하는 경우가 많습니다.

약어 설명(ASC):
- **ASC**는 **Ability System Component**의 약자입니다.
- 한국어로는 "어빌리티 시스템 컴포넌트"이며, 어빌리티·어트리뷰트·이펙트·태그를 한 액터에서 관리하는 허브 역할을 합니다.
- 멀티플레이에서 복제 설정과 권한 규칙을 함께 다루는 경우가 많습니다.

---

### 9-2 게임플레이 어빌리티

- **Gameplay Ability**는 스킬·구르기·조준 같은 **능력 단위**입니다. **Cost/Cooldown**을 태그·이펙트로 표현할 수 있습니다.
- **Activate** 시점에 애니메이션 몽타주 재생·타겟팅 등을 묶어 실행하기 좋습니다.
- **Ability Task**로 지연·대기·이동 같은 비동기 흐름을 안전하게 이어 붙일 수 있습니다.

---

### 9-3 게임플레이 어트리뷰트

- **Attribute Set**은 HP, MP, 공격력 등 **수치 상태**를 정의합니다. **Base / Current**와 **클램프** 규칙이 중요합니다.
- **GE(Gameplay Effect)**가 들어올 때 Base만 바꿀지 Current만 바꿀지에 따라 **버프·디버프 해석**이 달라집니다.
- UI는 보통 **Current** 값을 읽어 체력바를 갱신합니다.

---

### 9-4 게임플레이 이펙트

- **Gameplay Effect**는 일시적·지속적 변화를 줍니다. **Instant, Duration, Infinite** 유형이 있습니다.
- **Modifier**로 Attribute에 덧셈·곱셈 등 연산을 가하고, **Stacking** 규칙으로 중첩 버프를 제어합니다.
- **Execution Calculation**으로 복잡한 데미지 식을 C++ 또는 블루프린트에서 구현할 수 있습니다.

---

### 9-5 게임플레이 태그

- **Gameplay Tag**는 계층형 라벨(`Ability.Skill.Fire`)로 **상호 배타·요구 조건**을 선언적으로 묶습니다.
- **Granted Tags**와 **Remove Tags**로 스턴·침묵 같은 상태를 일관되게 켜고 끕니다.
- **Activation Required Tags**(활성화에 **필요한** 태그)와 **Activation Blocked Tags**(활성화를 **막는** 태그)로 "지금 이 스킬을 켜도 되는지"를 검사해, 위 **Granted Tags**로 걸린 스턴·침묵과 연동하기 쉽습니다.

---

### 9-6 게임플레이 큐

- **Gameplay Cue**는 **이펙트·사운드·카메라 쉐이크** 등 **표현 계층**을 태그 기반으로 실행합니다(로직과 분리).
- **Local Predicted** 등 실행 맥락을 나누면 클라이언트 예측과 서버 확정을 맞추기 쉽습니다.
- 로직(데미지 계산)과 표현(이펙트)을 나누면 **멀티플레이 디버깅**이 단순해집니다.

| 구성요소 | 역할 한 줄 |
|----------|------------|
| ASC | 능력·속성·이펙트 허브 |
| Ability | 플레이어 행동 단위 |
| Attribute | 수치 상태 |
| Effect | 속성/태그 변화 규칙 |
| Tag | 분류·조건·충돌 제어 |
| Cue | 비주얼·청각 표현 |

예시: GAS 켜기(재현 절차)
1. **Edit → Plugins**에서 **Gameplay Abilities**를 활성화하고 에디터를 재시작한다.
2. 캐릭터(또는 Pawn)에 **Ability System Component**를 추가한다.
3. C++ 또는 블루프린트에서 **Attribute Set**을 만들고 HP를 정의한다.
4. **Gameplay Effect**로 HP를 깎는 인스턴트 이펙트를 적용해 값이 변하는지 확인한다.

절차 해석:
- ASC가 없으면 어빌리티·속성·태그가 **붙을 대상**이 없어 GAS 흐름이 시작되지 않습니다.

오류 방지(플러그인·컴포넌트):
1. **언제**: **Gameplay Abilities** 플러그인을 켠 뒤에도 ASC 관련 노드·클래스가 보이지 않거나, 빌드만 실패할 때.
2. **왜**: 플러그인 변경 후 **에디터 재시작**이 필요하거나, 모듈 의존성(C++ 프로젝트)이 빌드 설정에 반영되지 않았을 수 있습니다.
3. **해결**: 플러그인 활성화 후 재시작을 확실히 하고, C++ 사용 시 `.Build.cs`에 `GameplayAbilities` 등 필요 모듈을 추가한 뒤 **프로젝트 파일 재생성**까지 검토합니다.

연습문제:
1. 문제: "스턴 상태에서는 스킬 입력 무시"를 태그만으로 표현한다면 어떤 태그 이름 규칙이 좋은지 예를 드세요.
   - 입력: 해당 없음
   - 출력: 태그 예 2개 이상(계층 포함)
   - 조건: `State.Debuff.Stun` 형태 권장
2. 문제: Gameplay Cue에 게임 규칙(데미지 계산)을 넣으면 왜 안 좋은지 적으세요.
   - 입력: 해당 없음
   - 출력: 이유 2문장
   - 조건: 멀티플레이 신뢰성 또는 재사용성 중 하나
3. 문제: **Instant**와 **Duration** GE를 각각 한 줄로 정의하고, 포션 회복에는 어느 쪽이 자연스러운지 고르세요.
   - 입력: 해당 없음
   - 출력: 정의 2줄 + 선택 + 이유
   - 조건: Duration은 "시간이 지나면 원상복귀" 예시 포함

정답 포인트:
- GAS = 능력+수치+효과+태그+표현 분리; 멀티플레이와 확장에 강함.
