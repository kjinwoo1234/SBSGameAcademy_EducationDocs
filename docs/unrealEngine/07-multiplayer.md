# Chapter 07 멀티플레이

## 학습 목표
- 서버 권한과 복제(Replicate)의 개념을 구분한다.
- RPC 종류와 사용 시점을 표로 정리할 수 있다.

## 세부 주제
- 서버-클라이언트 모델, 리슨/데디/클라이언트, RPC, Replication, FPS 실습 개요

## 실습 체크리스트
- **Play** 드롭다운에서 **Number of Players**를 2로 설정해 PIE 멀티 테스트를 한다.
- 변수 **Replicated**와 **RepNotify** 차이를 문서로 한 줄씩 정리한다.

## 본문

### 7-1 서버-클라이언트 모델

- **서버**가 게임 상태의 **권한 있는 소스**인 경우가 많습니다. 클라이언트는 입력을 보내고 결과를 표시합니다.
- 치트·불일치를 줄이려면 **중요 판정은 서버에서** 하는 것이 안전합니다.
- **Authority**(권한)가 있는 액터만 특정 복제·RPC 조합을 안전하게 수행할 수 있는 경우가 많습니다.

---

### 7-2 리슨 서버, 데디케이트 서버, 클라이언트

| 유형 | 설명 |
|------|------|
| Listen Server | 플레이어 중 한 명이 서버 역할까지 겸함 |
| Dedicated Server | 플레이 전용 서버, UI 없음 |
| Client | 서버에 연결만 하는 플레이어 인스턴스 |

---

### 7-3 RPC란?

약어 설명(RPC):
- **RPC**는 **Remote Procedure Call**의 약자입니다.
- 한국어로는 "원격 프로시저 호출"이며, 다른 네트워크 역할의 머신에서 함수를 실행하도록 요청하는 메커니즘입니다.
- 복제만으로는 부족한 **일회성 이벤트**(발사 요청, 이펙트 재생)를 역할에 맞게 나누기 위해 씁니다.
- 언리얼에서는 **Server / Client / Multicast** RPC로 구분합니다.

| 종류 | 용도 예시 |
|------|-----------|
| Server RPC | 클라이언트가 "발사 요청"을 서버에 전달 |
| Client RPC | 서버가 특정 클라이언트 UI만 갱신 |
| Multicast RPC | 모든 클라이언트에서 이펙트·사운드 재생 |

---

### 7-4 Replication이란?

용어 설명(Replication):
- **Replication**(복제)은 멀티플레이에서 서버가 가진 상태를 **네트워크를 통해 클라이언트 쪽 액터/변수에 맞추는** 메커니즘입니다.
- 한국어 자료에서는 "리플리케이션"으로도 많이 쓰이며, **치트 방지**와 **모든 클라이언트의 일관된 화면**을 위해 씁니다.
- **RPC**(함수 호출 전달)와 달리, 복제는 **변경된 프로퍼티를 주기적·필요 시 동기화**하는 쪽에 가깝습니다.

- **Replicated** 변수, **RepNotify**(값 변경 시 콜백)를 적절히 선택합니다.
- **Actor / Component**의 **Replication** 설정과 **Net Update Frequency**가 대역폭과 연관됩니다.

---

### 7-5 (실습) 멀티플레이 FPS 만들기

- **캐릭터·무기**는 소유권(Owner)과 **Server RPC**로 발사를 검증합니다.
- **히트 스캔**은 서버에서 Line Trace로 확정하는 패턴이 흔합니다.
- 클라이언트는 **예측(Prediction)**으로 총구 화염·반동만 먼저 보여 주고, 서버 결과가 오면 **보정**하는 식으로 지연을 숨깁니다.

예시: PIE 2인 플레이(재현 절차)
1. 툴바 **Play** 옆 드롭다운에서 **Number of Players**를 2로 설정한다.
2. **Net Mode**를 **Play As Listen Server** 등 프로젝트에 맞게 고른다.
3. 실행 후 두 뷰에서 캐릭터 이동이 서로 보이는지 확인한다.
4. **Show → Network** 관련 통계(버전에 따라 메뉴명 상이)로 패킷 흐름을 대략 본다.

절차 해석:
- PIE 멀티는 **로컬 루프백**이라 실제 온라인 지연·NAT 문제와는 다를 수 있습니다.

연습문제:
1. 문제: "총알 이펙트만 Multicast, 데미지는 Server에서만" 패턴의 이유를 한 문장으로 쓰세요.
   - 입력: 해당 없음
   - 출력: 이유 2문장 이내
   - 조건: 치트·불일치 키워드 중 하나 포함
2. 문제: Listen Server에서 플레이어 0번만 액터 소유권이 다른 이유를 추론해 보세요.
   - 입력: 해당 없음
   - 출력: 추론 문단 3문장 이내
   - 조건: "호스트" 개념 언급
3. 문제: **Replicated**와 **RepNotify**를 각각 한 줄로 정의하고, UI 갱신에는 어느 쪽이 더 잘 맞는지 적으세요.
   - 입력: 해당 없음
   - 출력: 정의 2줄 + 선택 + 이유 1줄
   - 조건: RepNotify는 "값 변경 시 함수" 점을 넣을 것

정답 포인트:
- 권한 = 서버 우선; RPC = 역할별 호출; Replication = 상태 동기화.
