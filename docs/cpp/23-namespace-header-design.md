# Chapter 23 네임스페이스와 헤더 설계

## 학습 목표
- 네임스페이스로 이름 충돌을 방지하는 방법을 익힌다.
- 헤더/소스 분리 원칙과 `#pragma once` 기반 중복 include 방지 방법을 이해한다.

## 세부 주제
### 23-1 네임스페이스 기초
- 이름 충돌 방지

### 23-2 헤더 설계 원칙
- 선언/정의 분리

### 23-3 `#pragma once`와 의존성 관리
- 중복 포함 방지

## 실습 체크리스트
- `math` 모듈을 헤더/소스로 분리하고 네임스페이스를 적용한다.

## 본문

### 23-1 네임스페이스 기초
- 대규모 코드에서는 같은 이름 함수/클래스가 충돌할 수 있습니다.
- 네임스페이스는 이름 충돌을 막아 모듈 경계를 명확히 합니다.

### 23-2 헤더 설계 원칙
- 헤더(`.h`)에는 선언, 소스(`.cpp`)에는 정의를 둡니다.
- 구현 세부사항을 숨기면 컴파일 의존성과 변경 영향을 줄일 수 있습니다.

### 23-3 `#pragma once`와 의존성 관리
- 헤더 중복 include 방지는 먼저 `#pragma once` 방식으로 이해하면 가장 직관적입니다.
- `#pragma once`는 헤더 파일을 한 번만 포함하도록 컴파일러에 지시해 재정의 오류를 줄입니다.
- 필요한 헤더만 포함하고, 과도한 include는 빌드 시간을 늘립니다.

재정의 오류가 언제 생기나:
- 같은 헤더가 한 번의 컴파일 단위에서 두 번 이상 포함되면, 함수 선언/클래스 정의/상수 정의가 중복 해석될 수 있습니다.
- 특히 헤더 A가 헤더 B를 포함하고, 소스 파일이 다시 헤더 B를 직접 포함하는 경우처럼 "간접 + 직접 include"가 겹칠 때 자주 발생합니다.

왜 `#pragma once` 또는 `#ifndef`가 필요한가:
- 두 방식 모두 "이 헤더 내용은 한 번만 처리"하게 만들어 중복 선언/정의로 인한 컴파일 오류를 방지합니다.
- 즉, 문제의 원인은 include 자체가 아니라 "같은 헤더 본문이 여러 번 펼쳐지는 것"이고, 이 두 방법은 그 원인을 차단하는 안전장치입니다.

`#pragma once` vs `#ifndef` 비교:
| 방식 | 사용 용도 | 장점 | 단점 |
|---|---|---|---|
| `#pragma once` | 헤더 중복 include 방지(입문 우선) | 코드가 짧고 실수 가능성이 낮음 | 표준 전처리 지시문은 아님(대부분 컴파일러는 지원) |
| `#ifndef ... #define ... #endif` | 헤더 중복 include 방지(전통 방식) | 표준 전처리 문법 기반, 이식성 설명에 유리 | 매크로 이름 충돌/오타 관리 필요 |

용어 설명:
| 용어 | 의미 |
|---|---|
| 네임스페이스(namespace) | 식별자 충돌을 막기 위한 이름 영역 |
| include guard | `#ifndef/#define/#endif`로 중복 include를 막는 전처리 패턴 |

헤더 설계 체크:
| 항목 | 권장 방식 |
|---|---|
| 선언 위치 | 헤더 |
| 정의 위치 | 소스 |
| 중복 포함 방지 | `#pragma once` 먼저 학습, `#ifndef` 방식도 함께 이해 |

예시 코드:
```cpp
// math_util.h
#pragma once
namespace math_util {
int add(int a, int b);
}
```

`#ifndef` 보조 예시(한 번만 참고):
```cpp
// math_util.h
#ifndef MATH_UTIL_H
#define MATH_UTIL_H
namespace math_util {
int add(int a, int b);
}
#endif
```

중복 include 상황 예시(오류 원인):
```cpp
// a.h
#pragma once
#include "b.h"

// b.h  (중복 방지 장치가 없다고 가정)
class Item { };

// main.cpp
#include "a.h"
#include "b.h"  // b.h가 간접+직접으로 두 번 들어와 재정의 위험
```

예시 설명:
- 위 구조에서 `b.h`에 중복 방지 장치가 없으면 `Item`이 두 번 정의되어 재정의 오류가 발생할 수 있습니다.
- `b.h`에 `#pragma once` 또는 `#ifndef`를 넣으면 두 번째 include는 무시되어 오류를 방지합니다.

```cpp
// math_util.cpp
#include "math_util.h"
namespace math_util {
int add(int a, int b) { return a + b; }
}
```

예상 결과:
```text
모듈 경계가 명확해지고 이름 충돌 가능성이 줄어든다.
```

코드 해석:
- `math_util::add`처럼 이름 공간을 분리해 함수 충돌을 예방합니다.
- 헤더에는 함수 "선언"만 두고 구현은 `.cpp`에서 제공합니다.
- `#pragma once`는 코드가 간결해 초보자가 중복 include 방지 개념을 잡기 좋습니다.
- `#ifndef` 방식은 기존 코드 해석/유지보수 상황에서 읽을 수 있도록 한 번만 익혀둡니다.
