# Chapter 15 도전! OOP 미니 프로젝트

## 학습 목표
- 클래스 분할과 책임 분리 설계를 적용한다.
- 상속/다형성을 작은 프로젝트에 통합한다.

## 세부 주제
### 15-1 프로젝트 요구사항 정리
- 기능 목록과 클래스 후보 도출

### 15-2 클래스 역할 분리
- 데이터 모델, 서비스, UI 분리

### 15-3 다형성 확장
- 공통 인터페이스 기반 기능 추가

## 실습 체크리스트
- 최소 3개 클래스 이상으로 미니 프로젝트를 구조화한다.

## 본문

### 15-1 프로젝트 요구사항 정리
- 먼저 "무엇을 만들지"를 기능 목록으로 분리해야 설계가 쉬워집니다.
- 예: 캐릭터 생성, 전투 로그 출력, 종료 저장.

약어 설명(OOP):
- **OOP**는 **Object-Oriented Programming**의 약자입니다.
- 한국어로는 "객체지향 프로그래밍"이며, 데이터와 동작을 객체 단위로 묶어 설계하는 방식입니다.
- 이 챕터에서는 OOP를 "클래스 책임 분리와 협력 구조 설계" 관점에서 적용합니다.

### 15-2 클래스 역할 분리
- 한 클래스가 너무 많은 일을 하지 않도록 역할을 나눕니다.
- 예: `Character`(데이터), `BattleSystem`(규칙), `Game`(흐름 제어).

### 15-3 다형성 확장
- 공통 기반 타입(`Enemy`)을 두면 적 타입 추가가 쉬워집니다.
- 새 적을 추가해도 전투 루프 코드 변경을 최소화할 수 있습니다.

용어 설명:
| 용어 | 의미 |
|---|---|
| 결합도(coupling) | 클래스끼리 얼마나 강하게 서로 의존하는지 나타내는 정도 |
| 단일 책임(SRP) | 한 클래스가 한 가지 역할에 집중해야 한다는 설계 원칙 |

설계 체크 표:
| 항목 | 확인 질문 |
|---|---|
| 단일 책임 | 이 클래스가 한 가지 역할만 수행하는가? |
| 결합도 | 다른 클래스 내부 구현에 과도하게 의존하는가? |
| 확장성 | 새 타입 추가 시 기존 코드 수정이 최소인가? |

예시 코드(구조 스케치):
```cpp
class Character { /* hp, atk, name */ };
class Enemy { public: virtual int damage() const = 0; };
class Slime : public Enemy { /* ... */ };
class BattleSystem { /* fight(Character&, Enemy&) */ };
class Game { /* run() */ };
```

예상 결과:
```text
클래스 역할이 분리되어 기능 추가 시 수정 범위가 줄어든다.
```

코드 해석:
- 이 챕터는 구현보다 "설계 구조"를 먼저 고정하는 단계입니다.
- 기반 인터페이스를 잡아두면 파생 클래스 확장이 쉬워집니다.
- `Game`은 흐름, `BattleSystem`은 규칙처럼 책임을 나누는 것이 핵심입니다.

연습문제:
1. 문제 1 - 클래스 다이어그램 만들기
   - 문제: 콘솔 RPG의 클래스 4개 이상을 정의하고 책임을 한 줄씩 작성하세요.
   - 입력: 없음
   - 출력: 클래스 목록/책임 표
   - 조건: 상속 관계 1개 이상 포함
2. 문제 2 - 전투 루프 구현
   - 문제: `Player`와 `Enemy`가 번갈아 공격하는 루프를 구현하세요.
   - 입력: 초기 체력/공격력
   - 출력: 턴별 전투 로그
   - 힌트: 전투 규칙은 별도 함수로 분리

정답 포인트:
- 구현 전에 요구사항/역할을 먼저 정리
- 클래스는 책임 단위로 나누고 인터페이스로 연결
